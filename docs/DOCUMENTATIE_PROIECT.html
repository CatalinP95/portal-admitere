1. Prezentare generală

Portal Admitere este o platformă web destinată universităților pentru gestionarea completă a procesului de admitere și a cazării studențești în cămine. Aplicația permite candidaților să depună cereri de admitere online, secretariatului să gestioneze și să proceseze aceste cereri, iar administrației să configureze algoritmii de selecție și să monitorizeze întregul sistem.

2. Arhitectura sistemului

Aplicația urmează o arhitectură de microservicii cu separare clară a responsabilităților. Toate cererile din frontend trec printr-un API Gateway central care aplică autentificarea și rate limiting-ul înainte de a ruta spre serviciile corespunzătoare.

4. Microservicii

🔍 Discovery Server

:8761

Netflix Eureka Server — registru central pentru toate microserviciile. Fiecare serviciu se înregistrează la pornire și se poate descoperi reciproc prin nume logic (ex: lb://user-service). Dashboard vizual la /eureka/web.

⚙️ Config Server

:8888

Spring Cloud Config Server — servește configurații centralizate pentru toate microserviciile. JWT secret-ul și credențialele bazelor de date sunt externalizate aici, nu hardcodate în servicii. Suportă @RefreshScope pentru reîncărcare fără restart.

🚪 API Gateway

:8080

Spring Cloud Gateway — punct unic de intrare. Aplică JWT filter, rate limiting cu Redis, load balancing round-robin între instanțele user-service, și LoggingFilter pentru audit. Rutele sunt configurate prin application.yml.

👤 User Service

:8081 / :8083 (2 instanțe)

Serviciul principal — autentificare JWT, gestionare utilizatori și profiluri, anunțuri cu taguri, audit log MongoDB, chatbot AI. Expune Swagger UI la /swagger-ui.html și metrici JaCoCo la /coverage.

🎓 Admissions Service

:8082

Gestionarea procesului de admitere — cereri candidați, algoritm de selecție bazat pe medie bacalaureat, generare contracte PDF, comunicare prin RabbitMQ. Implementat de Rareș.

🏠 Dormitory Service

:8084

Gestionarea cazării în cămine — cereri cazare, algoritm alocare locuri bazat pe medie și distanță, administrare camere și paturi. Implementat de Dragoș.

5. Frontend Angular 21

Frontend-ul este o Single Page Application construită cu Angular 21 standalone components, fără NgModules. Folosește signals pentru state management reactiv și sintaxa modernă @if / @for în template-uri.

Structura aplicației

frontend/src/app/
├── core/
│   ├── services/          — AuthService, UserService, AnnouncementService,
│   │                        ProfileService, ChatService
│   ├── models/            — Interfețe TypeScript: User, Announcement, Page<T>
│   ├── guards/            — AuthGuard, RoleGuard
│   └── interceptors/      — jwtInterceptor (adaugă Authorization header)
├── features/
│   ├── auth/              — Login, Register (split-screen animat)
│   ├── dashboard/
│   │   ├── users/         — CRUD utilizatori cu paginare, sortare, filtrare
│   │   ├── announcements/ — CRUD anunțuri cu taguri și search debounced
│   │   └── profile/       — Profil personal, schimbare parolă
│   └── errors/            — Pagini 404, 401 Access Denied
└── shared/
    ├── chatbot/            — Widget flotant RoboAdmis 🤖
    └── navbar/             — Navigare cu role badge

Configurare Spring Security

FuncționalitateImplementare
Password encodingBCryptPasswordEncoder (strength 10)
Token JWTjjwt 0.12.5 — HMAC-SHA256, secret din Config Server
Refresh TokenStocat în MySQL, invalidat la logout sau expirare
Autorizare endpoint-uri@PreAuthorize("hasRole('ADMIN')") pe metodele service/controller
CSRFCookieCsrfTokenRepository — cookie XSRF-TOKEN lizibil de JavaScript
CORSCorsConfigurationSource bean — origini configurabile prin ${ALLOWED_ORIGINS}
SessionSTATELESS — fără sesiuni server-side, doar JWT

CSRF în producție: În deployment cross-origin (Vercel ↔ Railway), cookie-ul XSRF-TOKEN nu traversează domenii diferite din cauza politicii SameSite a browserelor. Deoarece autentificarea se face exclusiv prin header Authorization: Bearer (nu cookie), CSRF nu e necesar cu JWT stateless — protecția este ignorată în producție prin ignoringRequestMatchers("/**"), dar infrastructura rămâne în cod.

9. Design Patterns

PatternUnde este aplicat
Strangler FigMigrare graduală din aplicația monolitică în microservicii — funcționalitățile sunt înlocuite progresiv fără a opri sistemul existent
RepositoryFiecare entitate JPA are propriul repository (UserRepository, AnnouncementRepository) care izolează accesul la date față de logica de business
Circuit BreakerResilience4j protejează user-service când admissions-service sau dormitory-service sunt indisponibile — cererile eșuate deschid circuitul și activează fallback-ul
StrategyAlgoritmul de admitere și cel de alocare cămin sunt strategii interschimbabile de ranking — formula de scor poate fi înlocuită fără a modifica controlerul
API GatewayPunct unic de intrare care aplică autentificare JWT, rate limiting și load balancing înainte de a ruta cererile
FallbackAdmissionsClientFallback și DormitoryClientFallback oferă răspunsuri degradate elegant când serviciile dependente nu răspund

13. Rulare locală

Prerequisite

Java 21, Maven 3.9+, Node.js 20+, Angular CLI 21, Docker Desktop

Pasul 1 — Infrastructură

docker compose up -d
# Pornește: MySQL :3306, MongoDB :27017, Redis :6379, RabbitMQ :5672, Prometheus :9090, Grafana :3000

Pasul 2 — Servicii (în ordine)

cd discovery-server  && mvn spring-boot:run   # Eureka :8761
cd config-server     && mvn spring-boot:run   # Config :8888
cd user-service      && mvn spring-boot:run   # User :8081
cd api-gateway       && mvn spring-boot:run   # Gateway :8080
# Opțional — admissions și dormitory:
cd admissions-service && mvn spring-boot:run  # :8082
cd dormitory-service  && mvn spring-boot:run  # :8084

Pasul 3 — Frontend

cd frontend && npm install && ng serve
# Aplicația: http://localhost:4300

Pasul 4 — Demo Load Balancing (opțional)

.\start-second-instance.ps1
# Pornește instanța 2 pe :8083, auto-download Maven dacă lipsește din PATH
# Eureka afișează: USER-SERVICE UP(2) localhost:8081, localhost:8083
# Header X-Instance-Port alternează între 8081 și 8083
