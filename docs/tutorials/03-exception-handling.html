🚨 Exception Handling

Gestionarea excepțiilor specifice per entitate cu @RestControllerAdvice

Problema: excepții generice

Fără o strategie clară, Spring returnează erori generice 500 Internal Server Error chiar și pentru cazuri simple cum ar fi un user inexistent. Clientul nu știe ce s-a întâmplat.

Soluție: Excepții dedicate per entitate + un handler global care le prinde și returnează răspunsuri JSON structurate cu cod HTTP corect.

Fluxul de tratare a erorilor

Service aruncă UserNotFoundException
→
GlobalExceptionHandler prinde excepția
→
Returnează 404 + ErrorResponse JSON

ErrorResponse – structura răspunsului

Toate erorile returnează același format JSON, consistent pentru frontend.

// ErrorResponse.java
public class ErrorResponse {
    private final String code;       // ex: "USER_NOT_FOUND"
    private final String message;    // ex: "User not found: 99"
    private final LocalDateTime timestamp;  // momentul erorii

    public ErrorResponse(String code, String message) {
        this.code = code;
        this.message = message;
        this.timestamp = LocalDateTime.now();
    }

    // getters...
}

Răspuns JSON returnat clientului:

{
  "code": "USER_NOT_FOUND",
  "message": "User not found: 99",
  "timestamp": "2026-05-12T14:30:00"
}

Excepții specifice per entitate

UserNotFoundException

public class UserNotFoundException extends RuntimeException {

    // caută după ID numeric
    public UserNotFoundException(Long id) {
        super("User not found: " + id);
    }

    // caută după username string
    public UserNotFoundException(String username) {
        super("User not found: " + username);
    }
}

AnnouncementNotFoundException

public class AnnouncementNotFoundException extends RuntimeException {
    public AnnouncementNotFoundException(Long id) {
        super("Announcement not found: " + id);
    }
}

TagNotFoundException și AlgorithmException

public class TagNotFoundException extends RuntimeException {
    public TagNotFoundException(String name) {
        super("Tag not found: " + name);
    }
}

public class AlgorithmException extends RuntimeException {
    public AlgorithmException(String message) {
        super(message);
    }
}

GlobalExceptionHandler – handler-ul central

@RestControllerAdvice permite interceptarea excepțiilor din orice controller Spring și returnarea unui răspuns personalizat.

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleUserNotFound(UserNotFoundException e) {
        return ResponseEntity
            .status(404)
            .body(new ErrorResponse("USER_NOT_FOUND", e.getMessage()));
    }

    @ExceptionHandler(AnnouncementNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleAnnouncementNotFound(AnnouncementNotFoundException e) {
        return ResponseEntity
            .status(404)
            .body(new ErrorResponse("ANNOUNCEMENT_NOT_FOUND", e.getMessage()));
    }

    @ExceptionHandler(TagNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleTagNotFound(TagNotFoundException e) {
        return ResponseEntity
            .status(404)
            .body(new ErrorResponse("TAG_NOT_FOUND", e.getMessage()));
    }

    @ExceptionHandler(AlgorithmException.class)
    public ResponseEntity<ErrorResponse> handleAlgorithmException(AlgorithmException e) {
        return ResponseEntity
            .status(500)
            .body(new ErrorResponse("ALGORITHM_ERROR", e.getMessage()));
    }

    // prinde orice altă excepție neașteptată
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneric(Exception e) {
        return ResponseEntity
            .status(500)
            .body(new ErrorResponse("INTERNAL_ERROR", "An unexpected error occurred"));
    }
}

Cum aruncăm excepțiile în Service

// UserService.java
public UserDto findById(Long id) {
    return userRepository.findById(id)
        .map(UserDto::from)
        .orElseThrow(() -> new UserNotFoundException(id));
}

// AnnouncementService.java
public AnnouncementDto update(Long id, AnnouncementRequest request) {
    Announcement announcement = announcementRepository.findById(id)
        .orElseThrow(() -> new AnnouncementNotFoundException(id));
    // ... actualizare...
}

Avantaje față de excepții generice:
✓ Frontend primește cod HTTP și mesaj clar
✓ Logging consistent (poți logga la nivel de handler)
✓ Ușor de extins cu noi tipuri de erori
✓ Testele pot verifica tipul exact de excepție aruncat

Întrebări frecvente la examen

Î: Ce face @RestControllerAdvice și cum diferă de @ControllerAdvice?
@RestControllerAdvice = @ControllerAdvice + @ResponseBody. Interceptează excepțiile aruncate din orice controller Spring și permite returnarea unui răspuns JSON personalizat. Fără @ResponseBody, Spring ar încerca să randeze un view (Thymeleaf/JSP). Cu @RestControllerAdvice, metoda returnează direct obiectul Java serializat ca JSON.
Î: De ce excepțiile noastre extind RuntimeException și nu Exception?
RuntimeException este „unchecked" — compilatorul nu obligă metodele care o aruncă să o declare în semnătură (throws UserNotFoundException). Dacă am extinde Exception (checked), fiecare metodă care apelează userRepository.findById() și aruncă excepția ar trebui să declare throws UserNotFoundException în semnătură — cod verbos și inutil în aplicații Spring.
Î: Ce cod HTTP returnezi când o resursă nu există? Dar pentru date invalide?
404 Not Found — resursa cerută nu există (UserNotFoundException, AnnouncementNotFoundException). 400 Bad Request — datele trimise de client sunt invalide (lipsesc câmpuri obligatorii, format incorect). 422 Unprocessable Entity — structura JSON e corectă dar valorile sunt invalide (alternativă la 400). 500 Internal Server Error — erori neașteptate de server (AlgorithmException, NullPointerException).
Î: Cum funcționează @ExceptionHandler și cine îl apelează?
Când o metodă de controller (sau de service, propagată prin controller) aruncă o excepție, Spring interceptează excepția înainte de a trimite răspunsul HTTP. Caută în @RestControllerAdvice o metodă adnotată cu @ExceptionHandler(TipulExceptiei.class) care corespunde tipului excepției. Dacă găsește, o apelează; altfel caută pentru superclase. Metoda cu @ExceptionHandler(Exception.class) este fallback-ul pentru orice excepție neprinse.
Î: Ce structură are ErrorResponse și de ce includem timestamp?
ErrorResponse conține: code (identificator string pentru tipul erorii, ex: USER_NOT_FOUND), message (detalii human-readable, ex: User not found: 99), timestamp (LocalDateTime.now() — momentul erorii). Timestamp-ul ajută la corelarea erorii cu log-urile serverului și este util pentru debugging și audit.
Î: Cum verifici în teste că o excepție specifică este aruncată?
Se folosește assertThrows(TipExceptie.class, () -> metoda()) din JUnit 5. Exemplu: assertThrows(UserNotFoundException.class, () -> userService.findById(99L)). Dacă metoda nu aruncă excepția sau aruncă un tip diferit, testul eșuează. Poți și captura excepția: var ex = assertThrows(...) și apoi verifica mesajul cu assertTrue(ex.getMessage().contains("99")).
Î: Care e avantajul excepțiilor specifice față de if/else + ResponseEntity direct în controller?
Cu excepții specifice: separarea responsabilităților — service-ul nu știe de HTTP, aruncă doar excepții de business. DRY — tratarea erorii e o singură dată în GlobalExceptionHandler. Testabilitate — testele de service verifică excepțiile, independent de HTTP. Alternativa (return ResponseEntity.notFound() din service) cuplează stratul de business cu HTTP și face testele mai complexe.

Implementare pas cu pas — Exception Handling

Prerequisite: Microserviciul Spring Boot cu cel puțin un model și un service (Tutorial 00 + 02).

Pasul 1 — Creează clasa ErrorResponse

Creează exception/ErrorResponse.java — aceasta este structura JSON returnată la erori:

public class ErrorResponse {
    private String code;        // ex: "USER_NOT_FOUND"
    private String message;     // ex: "User not found: 99"
    private LocalDateTime timestamp;

    public ErrorResponse(String code, String message) {
        this.code = code;
        this.message = message;
        this.timestamp = LocalDateTime.now();
    }
    // getters
}
Pasul 2 — Creează excepții specifice

Creează câte un fișier pentru fiecare entitate principală în folderul exception/:

// exception/UserNotFoundException.java
public class UserNotFoundException extends RuntimeException {
    public UserNotFoundException(Long id) {
        super("User not found: " + id);
    }
    public UserNotFoundException(String username) {
        super("User not found: " + username);
    }
}
// exception/DuplicateEmailException.java
public class DuplicateEmailException extends RuntimeException {
    public DuplicateEmailException(String email) {
        super("Email already in use: " + email);
    }
}

Creează câte o excepție similară pentru fiecare entitate: AnnouncementNotFoundException, ProfileNotFoundException etc.

Pasul 3 — Creează GlobalExceptionHandler

Creează exception/GlobalExceptionHandler.java:

@RestControllerAdvice  // interceptează excepțiile din toți controller-ii
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleUserNotFound(UserNotFoundException e) {
        return ResponseEntity.status(404)
                .body(new ErrorResponse("USER_NOT_FOUND", e.getMessage()));
    }

    @ExceptionHandler(DuplicateEmailException.class)
    public ResponseEntity<ErrorResponse> handleDuplicateEmail(DuplicateEmailException e) {
        return ResponseEntity.status(409)
                .body(new ErrorResponse("DUPLICATE_EMAIL", e.getMessage()));
    }

    @ExceptionHandler(AccessDeniedException.class)
    public ResponseEntity<ErrorResponse> handleAccessDenied(AccessDeniedException e) {
        return ResponseEntity.status(403)
                .body(new ErrorResponse("ACCESS_DENIED", "Nu ai permisiunea pentru această acțiune"));
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldErrors().stream()
                .map(f -> f.getField() + ": " + f.getDefaultMessage())
                .collect(Collectors.joining("; "));
        return ResponseEntity.status(400).body(new ErrorResponse("VALIDATION_ERROR", msg));
    }

    @ExceptionHandler(Exception.class)  // catch-all pentru excepții neprevăzute
    public ResponseEntity<ErrorResponse> handleGeneral(Exception e) {
        return ResponseEntity.status(500)
                .body(new ErrorResponse("INTERNAL_ERROR", "Eroare internă a serverului"));
    }
}
Pasul 4 — Folosește excepțiile în Service

În loc să returnezi null sau Optional gol din service, aruncă excepțiile:

@Service
public class UserService {

    public UserDto findById(Long id) {
        User user = userRepository.findById(id)
                .orElseThrow(() -> new UserNotFoundException(id));  // aruncă excepție specifică
        return UserDto.from(user);
    }

    public UserDto create(CreateUserRequest request) {
        if (userRepository.existsByEmail(request.email())) {
            throw new DuplicateEmailException(request.email());
        }
        // ... creare user
    }
}

Controller-ul rămâne curat — nu gestionează erori:

@GetMapping("/{id}")
public ResponseEntity<UserDto> getById(@PathVariable Long id) {
    return ResponseEntity.ok(userService.findById(id));  // excepția e gestionată de GlobalExceptionHandler
}

Verificare: Pornește aplicația și trimite GET la http://localhost:8081/api/users/9999 (ID inexistent).

Răspunsul trebuie să fie 404 cu body:

{"code": "USER_NOT_FOUND", "message": "User not found: 9999", "timestamp": "2026-05-16T..."}

Fără GlobalExceptionHandler, Spring ar returna un HTML de eroare sau un JSON cu stack trace.
