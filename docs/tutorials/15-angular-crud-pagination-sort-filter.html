Angular CRUD, Paginare, Sortare & Filtrare

Child routing, Page<T>, debounced search, sortare pe coloane, polling cu RxJS

Ce s-a construit în Ziua 15

Ziua 15 adaugă două module funcționale complete în dashboard: Anunțuri și Utilizatori. Ambele sunt pagini standalone lazy-loaded, montate ca child routes sub /dashboard, cu CRUD complet, paginare server-side, sortare pe coloane și filtrare cu debounce.

  • Child routing cu RouterOutlet — layout + conținut separat

  • UsersComponent — CRUD, search username/email, filtru rol, sort coloane

1. Arhitectura child routing

Înainte de Ziua 15, /dashboard era o singură componentă cu navbar + conținut. Pentru a adăuga pagini noi fără a duplica navbar-ul, s-a restructurat în layout + child routes.

/dashboard
→
DashboardComponent (layout: navbar + RouterOutlet)
/dashboard
→
DashboardHomeComponent
/dashboard/announcements
→
AnnouncementsComponent
/dashboard/users
→
UsersComponent (ADMIN)
// app.routes.ts
{
  path: 'dashboard',
  canActivate: [authGuard],
  loadComponent: () => import('./features/dashboard/dashboard').then(m => m.DashboardComponent),
  children: [
    {
      path: '',
      loadComponent: () => import('./features/dashboard/home/home').then(m => m.DashboardHomeComponent)
    },
    {
      path: 'announcements',
      loadComponent: () => import('./features/dashboard/announcements/announcements').then(m => m.AnnouncementsComponent)
    },
    {
      path: 'users',
      canActivate: [roleGuard],
      data: { roles: ['ADMIN'] },
      loadComponent: () => import('./features/dashboard/users/users').then(m => m.UsersComponent)
    }
  ]
}

Componenta DashboardComponent devine un simplu shell: navbar cu link-uri active + <router-outlet>. Conținutul fiecărei pagini se încarcă lazy în outlet.

// dashboard.html (layout)
<nav class="navbar">
  <a routerLink="/dashboard" routerLinkActive="active"
     [routerLinkActiveOptions]="{ exact: true }" class="nav-link">Acasă</a>
  <a routerLink="/dashboard/announcements" routerLinkActive="active" class="nav-link">Anunțuri</a>
  <a *ngIf="isAdmin" routerLink="/dashboard/users" routerLinkActive="active" class="nav-link">Utilizatori</a>
</nav>
<main class="content">
  <router-outlet></router-outlet>
</main>

[routerLinkActiveOptions]="{ exact: true }" pe linkul Acasă este obligatoriu. Fără exact, Angular marchează ca active orice rută care începe cu /dashboard — inclusiv /dashboard/announcements.

2. Interfața Page<T> — Spring Data Pageable

Spring Data returnează un obiect Page cu câmpuri standard. Frontend-ul definește o interfață TypeScript generică aliniată exact cu răspunsul JSON.

// core/models/page.model.ts
export interface Page<T> {
  content:        T[];
  totalElements:  number;
  totalPages:     number;
  number:         number;   // pagina curentă, 0-indexed
  size:           number;
  first:          boolean;
  last:           boolean;
  empty:          boolean;
}

Backend-ul acceptă automat parametrii de paginare și sortare prin Pageable:

Parametru URLEfect
?page=0&size=10Prima pagină, 10 înregistrări
?sort=username,ascSortare după username crescător
?sort=createdAt,descSortare după dată descrescător
?page=2&size=10&sort=email,ascCombinat

3. Service cu parametri dinamici

Serviciile folosesc HttpParams pentru a construi query string-ul dinamic, adăugând parametri opționali doar când există valori.

// announcement.service.ts
getAll(
  page = 0,
  size = 9,
  tag?: string,
  sort = 'createdAt,desc',
  search?: string
): Observable<Page<Announcement>> {
  let params = new HttpParams()
    .set('page', page)
    .set('size', size)
    .set('sort', sort);
  if (tag)    params = params.set('tag', tag);
  if (search) params = params.set('search', search);
  return this.http.get<Page<Announcement>>(this.base, { params });
}

4. Sortare pe coloane — click pe header

Componenta menține starea sortării (sortField + sortDir) și construiește parametrul pentru backend la fiecare schimbare.

sortField = 'createdAt';
sortDir: 'asc' | 'desc' = 'desc';

get sortParam(): string {
  return `${this.sortField},${this.sortDir}`;
}

sort(field: string): void {
  if (this.sortField === field) {
    this.sortDir = this.sortDir === 'asc' ? 'desc' : 'asc';
  } else {
    this.sortField = field;
    this.sortDir = 'asc';
  }
  this.currentPage = 0;
  this.load();
}

În template, header-ul de coloană afișează o săgeată care indică direcția curentă:

<th class="sortable" (click)="sort('username')">
  Utilizator
  <span class="sort-icon" [class.active]="sortField === 'username'">
    <svg ...>
      <polyline *ngIf="sortField === 'username' && sortDir === 'asc'"
                points="18 15 12 9 6 15"/>  <!-- săgeată sus -->
      <polyline *ngIf="sortField === 'username' && sortDir === 'desc'"
                points="6 9 12 15 18 9"/>   <!-- săgeată jos -->
      <polyline *ngIf="sortField !== 'username'"
                points="8 9 12 5 16 9"/>    <!-- neutru -->
    </svg>
  </span>
</th>

5. Debounced search cu RxJS Subject

Fără debounce, fiecare tastă ar declanșa un request HTTP. Cu Subject + debounceTime(400), request-ul se face abia după 400ms de pauză în tastare.

import { Subject, debounceTime, distinctUntilChanged } from 'rxjs';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';

private searchSubject = new Subject<string>();
private destroyRef = inject(DestroyRef);

ngOnInit(): void {
  this.load();

  this.searchSubject.pipe(
    debounceTime(400),
    distinctUntilChanged(),
    takeUntilDestroyed(this.destroyRef)
  ).subscribe(() => {
    this.currentPage = 0;
    this.load();
  });
}

onSearch(term: string): void {
  this.searchTerm = term;
  this.searchSubject.next(term);
}

takeUntilDestroyed(destroyRef) — oprește automat subscription-ul când componenta e distrusă (navigare la altă pagină). Fără el, subscription-ul rămâne activ și poate cauza memory leaks sau request-uri inutile.

6. Polling automat cu timer()

Anunțurile se reîncarcă automat la fiecare 30 de secunde, astfel încât un utilizator care stă pe pagină vede anunțurile noi adăugate de admin fără să dea refresh.

import { timer } from 'rxjs';

// În ngOnInit, după load() inițial:
timer(30000, 30000).pipe(
  takeUntilDestroyed(this.destroyRef)
).subscribe(() => this.load());

timer(delay, interval) — primul argument e delay-ul înainte de primul tick, al doilea e intervalul de repetiție. Cu timer(30000, 30000): primul reload la 30s, apoi la fiecare 30s.

De ce nu setInterval? setInterval nu e integrat cu RxJS și trebuie curățat manual în ngOnDestroy. timer() + takeUntilDestroyed() se curăță automat și sunt composable cu alți operatori RxJS.

7. Backend — filtrare cu @Query JPQL

Spring Data JPA permite filtrare cu parametri opționali prin @Query JPQL. Trucul este (:param IS NULL OR condiție) — dacă parametrul e null, condiția este ignorată.

// UserRepository.java
@Query("SELECT u FROM User u WHERE " +
       "(:search IS NULL OR LOWER(u.username) LIKE LOWER(CONCAT('%', :search, '%')) " +
       " OR LOWER(u.email) LIKE LOWER(CONCAT('%', :search, '%'))) " +
       "AND (:role IS NULL OR u.role = :role)")
Page<User> findWithFilters(
    @Param("search") String search,
    @Param("role") Role role,
    Pageable pageable);

Serviciul transformă string-urile goale în null înainte de a apela repository-ul:

public Page<UserDto> findFiltered(String search, String roleStr, Pageable pageable) {
    String searchParam = (search == null || search.isBlank()) ? null : search;
    Role roleParam   = (roleStr == null || roleStr.isBlank()) ? null
                       : Role.valueOf(roleStr.toUpperCase());
    return userRepository.findWithFilters(searchParam, roleParam, pageable)
                          .map(UserDto::from);
}

Sortarea vine automat prin Pageable — nu trebuie specificată în @Query. Spring Data aplică ORDER BY bazat pe ?sort=field,direction din URL, indiferent de query-ul de filtrare.

8. Filtrare anunțuri cu JOIN pe tags

Anunțurile au o relație @ManyToMany cu taguri. Filtrarea după tag necesită JOIN și DISTINCT pentru a evita duplicatele (un anunț cu mai multe taguri ar apărea de mai multe ori fără DISTINCT).

@Query(
  value = "SELECT DISTINCT a FROM Announcement a LEFT JOIN a.tags t WHERE a.enabled = true " +
          "AND (:tag IS NULL OR t.name = :tag) " +
          "AND (:search IS NULL OR LOWER(a.title) LIKE LOWER(CONCAT('%', :search, '%')))",
  countQuery = "SELECT COUNT(DISTINCT a.id) FROM Announcement a LEFT JOIN a.tags t " +
               "WHERE a.enabled = true " +
               "AND (:tag IS NULL OR t.name = :tag) " +
               "AND (:search IS NULL OR LOWER(a.title) LIKE LOWER(CONCAT('%', :search, '%')))"
)
Page<Announcement> findWithFilters(
    @Param("tag") String tag,
    @Param("search") String search,
    Pageable pageable);

countQuery separat — Spring Data JPA nu poate deriva automat query-ul de numărare din query-uri cu DISTINCT + JOIN. Fără countQuery, paginarea returnează totalPages greșit.

9. ChangeDetectorRef — pattern obligatoriu

Un comportament specific acestui proiect: Angular nu detectează automat schimbările de stare când callback-urile HTTP se execută în afara zonei Angular. Soluția este cdr.detectChanges() explicit în next și error.

load(): void {
  this.loading = true;
  this.service.getAll(...).subscribe({
    next: p => {
      this.page = p;
      this.loading = false;
      this.cdr.detectChanges(); // ← fără asta, spinner-ul rămâne
    },
    error: () => {
      this.error = 'Eroare.';
      this.loading = false;
      this.cdr.detectChanges();
    }
  });
}

Simptom fără detectChanges: request HTTP returnează 200, datele ajung în componentă (this.page = p), dar UI-ul rămâne pe spinner la infinit. Adăugarea lui cdr.detectChanges() rezolvă imediat.

10. Pattern modal — submit fără race condition

Ordinea operațiilor în callback-ul next al submit-ului este critică: întâi închidem modalul și forțăm detectarea schimbării, apoi apelăm load(). Dacă inversăm ordinea, cdr.detectChanges() din load() s-ar executa înainte ca modalul să fie vizual închis.

op$.subscribe({
  next: () => {
    this.showModal = false;      // 1. ascunde modalul
    this.editingId = null;
    this.modalLoading = false;
    this.cdr.detectChanges();     // 2. render fără modal
    this.load();                   // 3. reîncarcă lista
  },
  error: err => {
    this.modalError = 'Eroare.';
    this.modalLoading = false;
    this.cdr.detectChanges();
  }
});

Întrebări frecvente

De ce e nevoie de countQuery separat în @Query cu DISTINCT?

Spring Data JPA derivă automat query-ul de numărare din cel principal, dar nu știe să genereze COUNT(DISTINCT) pentru query-uri complexe cu JOIN. Fără countQuery, ar număra rândurile din JOIN (cu duplicate), returnând totalElements și totalPages greșite.

De ce debounceTime(400) și nu debounceTime(300) sau 1000?

400ms e empiric optimul perceput ca "responsiv" fără să facă request la fiecare tastă. Sub 300ms utilizatorul simte că face prea multe request-uri. Peste 600ms percepe lagul. 400ms e valoarea standard folosită în majoritatea aplicațiilor enterprise.

De ce LEFT JOIN în loc de INNER JOIN pentru tags?

Cu INNER JOIN, anunțurile fără niciun tag ar dispărea din rezultate chiar dacă nu filtrăm după tag. LEFT JOIN păstrează toate anunțurile și condiția (:tag IS NULL OR t.name = :tag) le filtrează corect: dacă :tag e null, condiția e mereu adevărată, deci apare tot.

Ce face takeUntilDestroyed(destroyRef)?

Completează automat un Observable când componenta Angular e distrusă (navigare la altă pagină, *ngIf false). Fără el, un timer() sau Subject rămâne activ în memorie după ce componenta dispare. destroyRef se injectează cu inject(DestroyRef) și poate fi folosit în orice context (nu doar constructor).

Care e diferența între timer(30000, 30000) și interval(30000)?

interval(30000) emite primul tick după 30s. timer(30000, 30000) face exact același lucru, dar permite specificarea unui delay inițial diferit de interval. De exemplu, timer(5000, 30000) face primul reload la 5s, apoi la fiecare 30s. Pentru polling, timer e mai flexibil.

Cum funcționează paginarea când filtrele se schimbă?

Când utilizatorul schimbă un filtru (search, tag, rol), currentPage se resetează la 0 înainte de apelarea load(). Fără reset, dacă ești pe pagina 3 și schimbi filtrul, ar cere pagina 3 din noul set de rezultate — care poate avea mai puțin de 3 pagini, returnând un content: [] gol.

De ce child routes și nu pagini separate cu navbar duplicat?

Child routes cu RouterOutlet evită duplicarea navbar-ului în fiecare componentă. Navbar-ul e instanțiat o singură dată în DashboardComponent (layout) și persistă între navigări. Beneficiu suplimentar: Angular nu recreează componenta layout la navigare între child routes — starea userului (username, rol) e citită o singură dată.

De ce se folosește :search IS NULL OR ... în JPQL și nu COALESCE?

În JPQL, COALESCE(:search, '') nu funcționează consistent pentru comparații LIKE în toate implementările Hibernate. Pattern-ul (:param IS NULL OR condiție) este portabil, clar și funcționează corect — Hibernate îl optimizează la nivel de SQL când parametrul e null.

Pasul 2 — Creează Service cu HttpParams pentru paginare 
@Injectable({ providedIn: 'root' })
export class UserAdminService {
  private url = `${environment.apiUrl}/api/users`;

  constructor(private http: HttpClient) {}

  getUsers(params: {
    page: number, size: number,
    sort: string, direction: 'asc' | 'desc',
    search?: string
  }): Observable<Page<User>> {
    let httpParams = new HttpParams()
      .set('page', params.page)
      .set('size', params.size)
      .set('sort', `${params.sort},${params.direction}`);

    if (params.search) {
      httpParams = httpParams.set('search', params.search);
    }

    return this.http.get<Page<User>>(this.url, { params: httpParams });
  }
}
Pasul 3 — Creează Component cu search debounced și sortare 
@Component({ standalone: true, imports: [CommonModule, FormsModule], ... })
export class UsersComponent implements OnInit, OnDestroy {
  users: User[] = [];
  page = 0; size = 10; totalPages = 0;
  sortField = 'username'; sortDir: 'asc'|'desc' = 'asc';
  search = '';

  // Subject pentru debounce la search — emite la fiecare keystroke
  private searchSubject = new Subject<string>();
  private sub = new Subscription();

  constructor(private userService: UserAdminService) {}

  ngOnInit() {
    // debounceTime(400): așteaptă 400ms de pauză între taste înainte de a trimite requestul
    this.sub.add(
      this.searchSubject.pipe(debounceTime(400), distinctUntilChanged())
        .subscribe(() => { this.page = 0; this.load(); })
    );
    this.load();
  }

  load() {
    this.userService.getUsers({
      page: this.page, size: this.size,
      sort: this.sortField, direction: this.sortDir,
      search: this.search || undefined
    }).subscribe(data => {
      this.users = data.content;
      this.totalPages = data.totalPages;
    });
  }

  onSearchChange(val: string) {
    this.search = val;
    this.searchSubject.next(val);   // trimite la Subject, nu direct la server
  }

  sort(field: string) {
    if (this.sortField === field) {
      this.sortDir = this.sortDir === 'asc' ? 'desc' : 'asc';
    } else {
      this.sortField = field; this.sortDir = 'asc';
    }
    this.load();
  }

  ngOnDestroy() { this.sub.unsubscribe(); }
}
