⚡ AOP + API Gateway + Circuit Breaker

@Aspect logging automat, rate limiting, LoggingFilter, Circuit Breaker pe 2 servicii

Ordinea de pornire a microserviciilor

Microserviciile au un lanț de dependențe la startup. Dacă pornești în ordine greșită, serviciile pornesc cu configurație incompletă sau nu se pot înregistra în registry.

1
config-server — port 8888
Servește configurația (JWT secret, DB URL, expiry) pentru toate celelalte. Dacă nu e up la pornire, serviciile folosesc valorile locale (fallback optional:), dar proprietățile externalizate nu se încarcă.
↓
2
discovery-server (Eureka) — port 8761
Registry-ul de servicii. user-service și api-gateway se înregistrează în Eureka la pornire. Dacă Eureka nu e up, ele reîncearcă periodic, dar e curat să fie gata înainte.
↓
3
user-service — port 8081
Cere configurație din Config Server ✓ și se înregistrează în Eureka ✓. Trebuie să fie înregistrat în Eureka înainte ca api-gateway să îl poată descoperi și ruta trafic spre el.
↓
4
api-gateway — port 8080
Rutează la lb://user-service rezolvat prin Eureka. Dacă pornește înainte ca user-service să fie înregistrat, prima cerere poate returna 503. Pornind ultimul, are instanța disponibilă imediat.

Regulă simplă: Config → Registry → Servicii → Gateway. Fiecare pas depinde de cel anterior.

Ce este API Gateway și de ce e necesar?

Fără gateway, clientul (Angular, Postman) trebuie să știe adresa exactă a fiecărui microserviciu. Cu gateway, există un singur punct de intrare pentru toate requesturile.

Fără GatewayCu Gateway
localhost:8081/api/users → direct user-servicelocalhost:8080/api/users → gateway → user-service
localhost:8082/admissions → direct admissions-servicelocalhost:8080/admissions → gateway → admissions-service
localhost:8083/camin → direct dormitory-servicelocalhost:8080/camin → gateway → dormitory-service

Ce face gateway-ul în proiect

ResponsabilitateImplementare
Routinglb://user-service via Eureka — știe unde e fiecare serviciu
AutentificareJwtGatewayFilter — validează JWT înainte să ajungă la serviciu
Rate limitingRequestRateLimiter — max N requesturi/secundă per user
LoggingLoggingFilter — loghează orice request cu timing
CORSConfigurare globală pentru Angular (localhost:4200)
Load balancingPrefix lb:// distribuie traficul între instanțe multiple

Avantaj cheie: Fără gateway, fiecare microserviciu ar trebui să implementeze JWT validation, CORS, rate limiting individual — cod duplicat în fiecare serviciu. Cu gateway, aceste responsabilități transversale sunt centralizate într-un singur loc.

LoggingFilter – request/response logging în api-gateway

Un GlobalFilter care interceptează orice request care trece prin gateway și loghează: metoda HTTP, path, cod de răspuns și durata în ms.

@Component
public class LoggingFilter implements GlobalFilter, Ordered {

    private static final Logger log = LoggerFactory.getLogger(LoggingFilter.class);

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        long start = System.currentTimeMillis();
        String method = exchange.getRequest().getMethod().name();
        String path   = exchange.getRequest().getPath().value();

        return chain.filter(exchange).then(Mono.fromRunnable(() -> {
            long elapsed = System.currentTimeMillis() - start;
            int status = exchange.getResponse().getStatusCode().value();
            log.info("{} {} → {} ({}ms)", method, path, status, elapsed);
        }));
    }

    @Override
    public int getOrder() { return -1; } // -1 = rulează primul, înainte de orice alt filter
}

Output în consola api-gateway:

POST /auth/login → 200 (23ms)
GET  /api/announcements → 200 (12ms)
GET  /api/users → 403 (5ms)

De ce GlobalFilter și nu un simplu @Aspect? api-gateway e bazat pe Spring WebFlux (reactiv, non-blocking). Nu are servlet-uri clasice, deci Spring MVC interceptors nu funcționează. GlobalFilter este echivalentul reactiv al unui servlet filter.

⚠️ Ordinea rutelor în application.yml — regulă critică

Spring Cloud Gateway evaluează rutele în ordinea în care sunt definite în YAML și o folosește pe prima care se potrivește. O greșeală frecventă: dacă ruta cu JwtGatewayFilter include /auth/**, atunci /auth/login (care nu ar trebui să necesite JWT) este blocat cu 401 Unauthorized.

# ❌ GREȘIT — user-service-public nu va fi atins niciodată pentru /auth/login
routes:
  - id: user-service           # potrivire /auth/** → JwtFilter → 401 la login!
    predicates:
      - Path=/auth/**, /api/**
    filters:
      - name: JwtGatewayFilter
  - id: user-service-public    # nu ajunge aici niciodată
    predicates:
      - Path=/auth/login, /auth/register

# ✅ CORECT — ruta publică vine PRIMA, fără JWT
routes:
  - id: user-service-public    # PRIMA — prinde /auth/login, /register, /refresh
    predicates:
      - Path=/auth/login, /auth/register, /auth/refresh
  - id: user-service           # A DOUA — prinde /auth/logout și /api/** cu JWT
    predicates:
      - Path=/auth/logout, /api/**
    filters:
      - name: JwtGatewayFilter

Regulă: Rutele publice (fără autentificare) merg ÎNTOTDEAUNA înaintea rutelor protejate în application.yml. Gateway-ul nu are o logică de „cel mai specific câștigă" — câștigă primul care se potrivește.

Rate Limiting cu Redis – protecție împotriva DoS

Limitează câte requesturi poate face un client pe secundă. Configurat în application.yml al api-gateway, folosind Redis pentru a stoca contoarele.

1. Dependință în pom.xml (api-gateway)

<!-- Redis reactive — necesar pentru RequestRateLimiter din Gateway -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis-reactive</artifactId>
</dependency>

De ce -reactive? api-gateway rulează pe WebFlux (reactiv). spring-boot-starter-data-redis standard funcționează cu Servlet/blocking. Versiunea -reactive oferă un client Redis non-blocking compatibil cu WebFlux.

2. KeyResolver – pe ce se bazează limita

@Configuration
public class KeyResolverConfig {

    @Bean
    public KeyResolver userKeyResolver() {
        return exchange -> Mono.justOrEmpty(
            exchange.getRequest().getHeaders().getFirst("X-User-Id")
        ).defaultIfEmpty("anonymous");
        // limita e per userId sau "anonymous" dacă nu e autentificat
    }
}

3. Configurare în application.yml

spring:
  data:
    redis:
      host: localhost
      port: 6379
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 20   # tokeni adăugați/secundă
                redis-rate-limiter.burstCapacity: 40   # maxim tokeni simultan
                key-resolver: "#{@userKeyResolver}"
ParametruRuta /auth/** (publică)Ruta /api/** (protejată)Efect
replenishRate105Tokeni adăugați per secundă (rata normală)
burstCapacity2010Tokeni inițiali în bucket (maxim burst)

Algoritmul Token Bucket — cum funcționează exact

Rate limiting-ul folosește algoritmul Token Bucket, implementat în Redis:

Exemplu pentru ruta /api/** (burstCapacity=10, replenishRate=5):

Bucket începe cu 10 tokeni
Request 1  → consumă 1 token → rămân 9  → 200 OK
Request 2  → consumă 1 token → rămân 8  → 200 OK
...
Request 10 → consumă 1 token → rămân 0  → 200 OK
Request 11 → bucket gol      → rămân 0  → 429 Too Many Requests

Dacă aștepți 1 secundă → se adaugă 5 tokeni → poți face 5 requesturi noi

Valorile pentru demo vs producție: Postman Runner trimite requesturi secvențial (~200ms/request). Cu replenishRate: 5, în 3 secunde pentru 15 requesturi se refac 15 tokeni — bucket-ul nu se golește niciodată. Soluție: replenishRate: 1, burstCapacity: 5 — cu 1 token/secundă refill și request-uri la 200ms, bucket-ul se epuizează după primele 5. În producție valorile normale ar fi replenishRate: 20, burstCapacity: 40.

Când limita e depășită, gateway-ul returnează automat 429 Too Many Requests fără să ajungă la user-service. Headers utile în response:

X-RateLimit-Remaining: 0
X-RateLimit-Burst-Capacity: 10
X-RateLimit-Replenish-Rate: 5
X-RateLimit-Requested-Tokens: 1

@Aspect AOP – logging automat în user-service

AOP (Aspect-Oriented Programming) permite adăugarea de comportament transversal (logging, timing, security) fără a modifica codul existent. Un singur aspect interceptează automat toate metodele din package-ul service.

Controller apelează AnnouncementService.getAll()
→
@Around interceptează înainte de execuție
→
log "→ START AnnouncementService.getAll(..)"
→
pjp.proceed() — execuție reală
→
log "← END ... in 12ms"
@Aspect
@Component
public class ServiceLoggingAspect {

    private static final Logger log = LoggerFactory.getLogger(ServiceLoggingAspect.class);

    // Pointcut: interceptează TOATE metodele publice din package-ul service
    @Pointcut("execution(* com.campus.userservice.service.*.*(..))")
    public void serviceMethods() {}

    @Around("serviceMethods()")
    public Object logExecution(ProceedingJoinPoint pjp) throws Throwable {
        String sig = pjp.getSignature().toShortString();
        long start = System.currentTimeMillis();
        log.info("→ START {}", sig);
        try {
            Object result = pjp.proceed(); // execuția metodei reale
            log.info("← END {} in {}ms", sig, System.currentTimeMillis() - start);
            return result;
        } catch (Exception e) {
            log.error("✗ FAIL {} after {}ms: {}", sig, System.currentTimeMillis() - start, e.getMessage());
            throw e;
        }
    }
}
AdnotareRol
@AspectMarchează clasa ca aspect AOP
@ComponentSpring îl creează ca bean
@PointcutDefinește unde se aplică aspectul (expresie SpEL)
@AroundRulează în jurul metodei — poate executa cod înainte și după
pjp.proceed()Execută metoda originală și returnează rezultatul

Avantaj față de @Slf4j manual: Fără aspect, trebuie să adaugi log.info() în fiecare metodă din fiecare serviciu. Cu aspectul, zero modificări în servicii — orice metodă nouă adăugată în service.* este loggată automat.

Dependință necesară în pom.xml

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>

Circuit Breaker #2 – discovery-server

Primul Circuit Breaker (ziua 7) protejează apelurile spre admissions-service. Al doilea protejează apelul spre discovery-server (Eureka).

@Service
public class HealthCheckService {

    private final RestTemplate restTemplate = new RestTemplate();

    @CircuitBreaker(name = "discovery-server", fallbackMethod = "discoveryFallback")
    public String checkDiscovery() {
        return restTemplate.getForObject(
            "http://localhost:8761/actuator/health", String.class);
    }

    // apelat automat de Resilience4j când CB e OPEN
    public String discoveryFallback(Exception e) {
        log.warn("Discovery server unavailable: {}", e.getMessage());
        return "{\"status\": \"Discovery unavailable\"}";
    }
}

Configurare în application.yml

resilience4j:
  circuitbreaker:
    instances:
      admissions-service:     # CB #1 — din ziua 7
        slidingWindowSize: 5
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
      discovery-server:       # CB #2 — adăugat azi
        slidingWindowSize: 3
        failureRateThreshold: 50
        waitDurationInOpenState: 5s

Stările Circuit Breaker

StareComportamentTranziție
CLOSED Normal — requesturile trec, erorile sunt numărate → OPEN când failureRate ≥ 50% în ultimele 3 apeluri
OPEN Blocat — toate apelurile returnează fallback imediat → HALF_OPEN după 5 secunde
HALF_OPEN Testare — lasă 1 apel să treacă, verifică dacă serviciul e ok → CLOSED dacă reușește, → OPEN dacă eșuează

De ce 2 Circuit Breakers separați? Fiecare serviciu extern are proprii parametri de toleranță. admissions-service are slidingWindowSize: 5 (mai tolerant), discovery-server are slidingWindowSize: 3 (mai agresiv — Eureka trebuie să fie mereu disponibil).

Întrebări frecvente la examen

Î: De ce microserviciile se pornesc în ordinea config → discovery → servicii → gateway?
Este un lanț de dependențe la startup. Config Server trebuie să fie primul — toate serviciile îi cer configurația la pornire. Eureka al doilea — serviciile se înregistrează în el la pornire. user-service înaintea gateway-ului — gateway-ul rezolvă lb://user-service prin Eureka; dacă user-service nu e înregistrat, gateway-ul nu are unde să ruteze. Gateway ultimul — are nevoie de toți ceilalți gata.
Î: Ce este un @Pointcut și care este sintaxa expresiei pentru service methods?
@Pointcut definește unde se aplică aspectul — o expresie care selectează metode. execution(* com.campus.userservice.service.*.*(..)) se citește: execuție (execution) a oricărei metode (*) din orice clasă (*) din packetul service, cu orice argumente ((..)). Primul * = orice tip de return, al doilea * = orice clasă, al treilea * = orice metodă.
Î: Diferența dintre @Before, @After și @Around în AOP?
@Before — rulează ÎNAINTE de metodă, nu poate opri execuția. @After — rulează DUPĂ metodă (întotdeauna, indiferent de excepții). @AfterReturning — doar dacă metoda returnează normal. @AfterThrowing — doar dacă aruncă excepție. @Around — cel mai puternic: controlează complet execuția, poate modifica parametrii, rezultatul sau chiar să nu apeleze metoda originală (pjp.proceed() este opțional).
Î: Ce returnează gateway-ul când rate limit e depășit și cum se configurează?
Returnează automat 429 Too Many Requests cu header X-RateLimit-Remaining: 0. Se configurează în application.yml al gateway-ului sub filters cu name: RequestRateLimiter. replenishRate = tokeni regenerați pe secundă (rata normală). burstCapacity = maxim tokeni acumulați (permite burst-uri). Algoritmul folosit este Token Bucket — Redis stochează numărul de tokeni disponibili per key.
Î: De ce cu burstCapacity=40 și Postman Runner nu se obține 429, chiar dacă se trimit 25 de requesturi?
Algoritmul Token Bucket refacе tokeni continuu: cu replenishRate=20, se adaugă 20 tokeni/secundă. Postman Runner secvențial trimite ~1 request la 50ms → 25 requesturi durează ~1.25 secunde → în acest timp se refac 25 tokeni (20×1.25). Net: 25 consumați − 25 refăcuți = bucket nu se golește niciodată. Soluție pentru demo: scazi burstCapacity la 10 și replenishRate la 5 — bucket-ul se epuizează la al 11-lea request rapid. În producție valorile mai mari sunt corecte.
Î: Cum funcționează fallback-ul în @CircuitBreaker și care sunt cerințele metodei fallback?
Când CB-ul e în starea OPEN (sau metoda aruncă excepție), Resilience4j apelează automat metoda specificată în fallbackMethod. Cerințe: aceeași semnătură ca metoda principală + un parametru extra de tip Exception (sau Throwable). Exemplu: metoda principală returnează String → fallback returnează tot String și primește Exception e. Dacă semnătura nu corespunde, Resilience4j aruncă NoFallbackAvailableException.
Î: De ce api-gateway folosește spring-boot-starter-data-redis-reactive și nu varianta standard?
api-gateway este construit pe Spring WebFlux (stack reactiv, non-blocking, bazat pe Netty). spring-boot-starter-data-redis standard folosește Lettuce sau Jedis în mod blocking — incompatibil cu un event loop reactiv (blochează thread-ul). Varianta -reactive oferă ReactiveRedisTemplate cu operații asincrone (Mono/Flux) care nu blochează event loop-ul gateway-ului.
Î: Ce avantaj are @Aspect față de logarea manuală cu @Slf4j în fiecare serviciu?
@Slf4j manual: trebuie adăugat log.info() explicit în fiecare metodă — cod repetitiv, ușor de uitat, și dacă adaugi un serviciu nou trebuie să adaugi manual logarea. @Aspect: zero modificări în servicii — orice metodă nouă din service.* este loggată automat. Avantaj suplimentar: poți schimba comportamentul de logging (format, nivel, destinație) dintr-un singur loc fără să atingi serviciile. Dezavantaj: @Aspect nu funcționează pe metode private sau metode apelate intern în aceeași clasă (self-invocation problem).

Implementare pas cu pas — AOP, Gateway și Circuit Breaker

Prerequisite: user-service rulând (Tutorial 00-09). Discovery Server Eureka pornit pe 8761.

Pasul 1 — Adaugă @Aspect pentru logging automat în user-service

Adaugă dependința AOP în user-service/pom.xml:

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>

Creează config/LoggingAspect.java:

@Aspect
@Component
@Slf4j
public class LoggingAspect {

    // Pointcut: interceptează ORICE metodă publică din pachetul service
    @Pointcut("execution(* com.campus.userservice.service.*.*(..))")
    public void serviceMethods() {}

    @Around("serviceMethods()")
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        String methodName = joinPoint.getSignature().toShortString();
        long start = System.currentTimeMillis();

        try {
            Object result = joinPoint.proceed();   // execută metoda originală
            long duration = System.currentTimeMillis() - start;
            log.info("[AOP] {} completed in {}ms", methodName, duration);
            return result;
        } catch (Exception e) {
            log.error("[AOP] {} threw: {}", methodName, e.getMessage());
            throw e;
        }
    }
}

Acum toate metodele din service.* sunt logate automat — fără nicio modificare în servicii.

Pasul 2 — Creează modulul api-gateway

Generează proiect nou la start.spring.io: Gateway + Eureka Discovery Client + Spring Cloud Circuit Breaker. Port: 8080.

Configurează api-gateway/src/main/resources/application.yml:

server:
  port: 8080

spring:
  application:
    name: api-gateway
  cloud:
    gateway:
      routes:
        # Rutele publice FĂRĂ autentificare (trebuie ÎNAINTE de ruta protejată)

            - Path=/auth/login, /auth/register, /auth/refresh, /auth/csrf

Pasul 3 — Adaugă Circuit Breaker

Circuit Breaker protejează Gateway-ul când un serviciu downstream este indisponibil:

# În application.yml al gateway:
- id: admissions-service
  uri: lb://admissions-service
  predicates:
    - Path=/admissions/**
  filters:
    - name: CircuitBreaker
      args:
        name: admissionsCircuitBreaker
        fallbackUri: forward:/fallback/admissions
        # Dacă admissions-service nu răspunde → redirecționează la /fallback/admissions

Creează controller-ul de fallback în gateway:

@RestController
public class FallbackController {

    @GetMapping("/fallback/admissions")
    public ResponseEntity<Map<String, String>> admissionsFallback() {
        return ResponseEntity.status(503).body(
            Map.of("message", "Serviciul de admitere este temporar indisponibil")
        );
    }
}

4. Pornești user-service          # port 8081
5. Pornești api-gateway           # port 8080 (ultimul!)
