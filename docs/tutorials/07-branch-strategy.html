🌿 Branch Strategy

Strategia dev/main – dezvoltare pe dev, release pe main

De ce branch-uri separate?

Fără o strategie clară, echipa lucrează direct pe main și riscă să introducă cod neterminat sau buguri în versiunea "stabilă". Separarea dev / main protejează producția.

Regulă în proiect:
dev → unde se dezvoltă zilnic, cod work-in-progress
main → cod stabil, merge-uri la finalul unui milestone

Vizualizarea fluxului

main
 ← merge la release
dev
 ← commit-uri zilnice

Setup inițial – crearea branch-ului dev

# Creează branch-ul dev pornind din main
git checkout -b dev

# Publică branch-ul pe GitHub (prima dată)
git push -u origin dev

# Verifică că ești pe dev
git branch
  * dev     ← asteriscul = branch-ul curent
    main

Workflow zilnic pe dev

# 1. Asigură-te că ești pe dev
git checkout dev

# 2. Adu ultimele modificări din GitHub
git pull origin dev

# 3. Faci modificările tale, testezi
# ... cod, teste ...

# 4. Adaugi fișierele modificate
git add src/main/java/com/campus/...

# 5. Commit cu mesaj descriptiv
git commit -m "feat: day 9 — specific exceptions, @OneToMany#2, Pageable#3, rememberMe"

# 6. Push pe GitHub
git push origin dev

Merge dev → main la release

# 1. Switch pe main
git checkout main

# 2. Aduce ultimele modificări din main
git pull origin main

# 3. Merge din dev (--no-ff păstrează istoricul clar)
git merge dev --no-ff -m "release: merge dev → main after Day 10"

# 4. Push main pe GitHub
git push origin main

# 5. Revino pe dev pentru a continua lucrul
git checkout dev

Comenzi utile

# Vezi istoricul grafic al branch-urilor
git log --oneline --graph --all

# Verifică pe ce branch ești
git branch

# Verifică statusul fișierelor
git status

# Anulează ultimul commit (păstrează modificările)
git reset --soft HEAD~1

# Destagingizează toate fișierele
git restore --staged .

GitHub Desktop: Dacă după git reset --soft HEAD~1 nu vezi modificările în GitHub Desktop, rulează și git restore --staged . pentru a le destaging-ui. Vor apărea ca modificări nesalvate.

Credentials Windows: Dacă push-ul eșuează cu "Permission denied", mergi la Control Panel → Credential Manager → Windows Credentials și actualizează intrarea git:https://github.com cu username-ul corect (CatalinP95 în loc de CatalinP14).

De ce --no-ff la merge?

Fără --no-ff, Git face un "fast-forward" merge și istoricul arată ca și cum toate commit-urile au fost făcute direct pe main. Cu --no-ff, se creează un commit de merge explicit și e clar că a existat un branch separat.

Întrebări frecvente la examen

Î: Care este rolul branch-ului dev față de main în strategia noastră?
dev = branch-ul de lucru zilnic, conține cod work-in-progress, poate fi instabil. Toți commit-urile zilnice merg pe dev. main = branch-ul stabil, reprezintă versiunea de „release". Merge-ul din dev în main se face doar la finalul unui milestone (ex: după ziua 10), cu comanda git merge dev --no-ff. Nimeni nu commit-uiește direct pe main.
Î: Ce face git merge --no-ff și de ce e preferabil față de fast-forward?
--no-ff (no fast-forward) forțează crearea unui commit explicit de merge, chiar dacă Git ar putea face fast-forward. Avantaj: istoricul Git arată clar că a existat un branch separat și când s-a făcut merge-ul. Cu fast-forward, commit-urile din dev apar ca și cum au fost făcute direct pe main — istoricul devine liniar și pierzi contextul „am lucrat pe un branch separat". În proiect: git merge dev --no-ff -m "release: merge dev → main after Day 10".
Î: Ce prefixe de commit se folosesc și când?
feat: — funcționalitate nouă (ex: feat: add JWT auth). fix: — corectare bug (ex: fix: FK constraint in tests). refactor: — restructurare fără funcționalitate nouă. test: — adăugare/modificare teste. chore: — configurare, build, dependințe (ex: chore: add docker-compose.yml). docs: — documentație. release: — merge la release. Convenția Conventional Commits face istoricul parsabil automat pentru changelog-uri.
Î: Care sunt pașii workflow-ului zilnic pe branch dev?
1. git checkout dev — asigură-te că ești pe dev. 2. git pull origin dev — sincronizează cu GitHub (important dacă lucrezi în echipă). 3. Faci modificările și testezi. 4. git add <fișiere-specifice> — staged doar ce e relevant. 5. git commit -m "feat: ..." — commit cu mesaj descriptiv. 6. git push origin dev — publici pe GitHub. Nu folosi git add . pentru a evita commit-uri accidentale de fișiere sensibile.
Î: Ce face git reset --soft HEAD~1 și când e util?
git reset --soft HEAD~1 „anulează" ultimul commit, dar păstrează modificările staged (pot fi recommit-uite). Util când ai scris un mesaj greșit sau ai uitat să incluzi un fișier. Alternativă: git reset --mixed HEAD~1 — modificările rămân unstaged. git reset --hard HEAD~1 — modificările sunt pierdute complet (folosești doar cu mare atenție). Nu folosi niciodată reset pe commit-uri deja push-uite pe branch-uri partajate.
Î: Cum publici un branch local nou pe GitHub pentru prima dată?
git push -u origin dev — flag-ul -u (set-upstream) leagă branch-ul local de cel remote. Ulterior, poți folosi simplu git push și git pull fără a specifica origin dev. Fără -u, Git nu știe la care branch remote să pusheze și îți cere să specifici manual de fiecare dată. git branch -vv afișează ce branch local e legat de ce remote.
Î: Cum vizualizezi istoricul grafic al branch-urilor în terminal?
git log --oneline --graph --all afișează un arbore ASCII al tuturor branch-urilor: * abc123 (HEAD -> dev) feat: add exception handling și merge commit-urile cu linii de conexiune. --oneline = câte o linie per commit, --graph = linii ASCII pentru structura branch-urilor, --all = include și branch-urile remote (origin/main, origin/dev).

Implementare pas cu pas — Workflow cu branch-uri

Prerequisite: Git instalat, cont GitHub creat, repository creat pe GitHub (fără fișiere inițiale).

Pasul 1 — Configurare inițială Git 
# O singură dată, setează identitatea
git config --global user.name "Prenume Nume"
git config --global user.email "email@example.com"

# În folderul proiectului
git init
git branch -M main
git remote add origin https://github.com/username/portal-admitere.git
Pasul 2 — Primul commit și creare branch dev 
git add .
git commit -m "feat: initial project structure"
git push -u origin main

# Creează branch-ul de dezvoltare
git checkout -b dev
git push -u origin dev
Pasul 3 — Workflow zilnic pe dev 
# Asigură-te că ești pe dev și actualizat
git checkout dev
git pull origin dev

# ... implementezi funcționalitatea ...

# Adaugă fișierele modificate (specific, nu git add .)
git add src/main/java/com/campus/userservice/api/UserController.java
git commit -m "feat: add user CRUD endpoints"
git push origin dev

Convenții pentru mesajele de commit: feat: funcționalitate nouă, fix: corecție bug, refactor: restructurare fără funcționalitate nouă, test: teste, docs: documentație.

Pasul 4 — Merge în main pentru release 
git checkout main
git pull origin main
git merge --no-ff dev -m "merge: integrate user service v1"
git push origin main

--no-ff forțează crearea unui merge commit vizibil în istoric. Fără el, Git aplatizează istoricul și nu se mai vede că modificările au venit din branch-ul dev.

Pasul 5 — Recuperare din greșeli comune 
# Anulează ultimul commit dar păstrează fișierele modificate
git reset --soft HEAD~1

# Revino la starea unui fișier din ultimul commit
git restore src/main/java/.../UserController.java

# Vizualizează istoricul grafic complet
git log --oneline --graph --all

Verificare: Pe GitHub trebuie să apară ambele branch-uri. Rulează local:

git log --oneline --graph --all
# Așteptat:
# *   a1b2c3 (HEAD -> main) merge: integrate user service v1
# |\
# | * d4e5f6 (dev) feat: add user CRUD endpoints
# |/
# * g7h8i9 feat: initial project structure
