Angular 19 – Setup, AuthService, Login & Register

Standalone components, feature modules, formulare reactive, CORS

Ce este Angular și de ce îl folosim

Angular este un framework front-end dezvoltat de Google care oferă o structură completă pentru aplicații web: routing, formulare reactive, HTTP client, change detection. Spre deosebire de React (bibliotecă), Angular vine cu toate aceste instrumente integrate.

În proiect folosim Angular ca single-page application (SPA) care comunică cu backend-ul exclusiv prin API-ul expus de API Gateway (port 8080). Browserul nu cunoaște microserviciile individuale.

withFetch() – de evitat: Angular 19 permite withFetch() pentru a folosi Fetch API în loc de XHR. Însă cu Fetch, răspunsurile 4xx (401, 400) pot apărea ca net::ERR_ABORTED în consolă și nu propagă corect eroarea la handler-ul din componentă. Soluție: nu folosi withFetch() în combinație cu error handling pe 4xx.

AuthService – Token Management

Token-ul JWT se salvează în localStorage (rememberMe = true) sau sessionStorage (sesiune curentă). La fiecare request, interceptorul citește token-ul și îl adaugă ca header.

@Injectable({ providedIn: 'root' })
export class AuthService {
  login(req: LoginRequest): Observable<AuthResponse> {
    return this.http.post<AuthResponse>(`${this.base}/login`, req).pipe(
      tap(res => this.saveTokens(res, req.rememberMe ?? false))
    );
  }

  logout(): Observable<void> {
    const headers = new HttpHeaders({
      'Refresh-Token': this.getRefreshToken() ?? ''
    });
    return this.http.post<void>(`${this.base}/logout`, {}, { headers }).pipe(
      tap(() => this.clearTokens())
    );
  }
}

De ce Refresh-Token în header? Backend-ul (/auth/logout) primește refresh token-ul prin header Refresh-Token, nu în body. Dacă nu trimiți header-ul, server-ul returnează 500.

Formulare reactive – Validator custom

Formularul de register are un validator la nivel de grup care verifică că cele două parole coincid:

function passwordMatchValidator(control: AbstractControl): ValidationErrors | null {
  const password = control.get('password')?.value;
  const confirm  = control.get('confirmPassword')?.value;
  return password && confirm && password !== confirm
    ? { passwordMismatch: true }
    : null;
}

// Aplicat pe grup, nu pe câmp individual:
this.form = this.fb.group({
  password:        ['', [Validators.required, Validators.minLength(8)]],
  confirmPassword: ['', Validators.required]
}, { validators: passwordMatchValidator });

markAllAsTouched(): Fără apel explicit, validările apar doar pe câmpurile atinse (blur). La submit, apelează this.form.markAllAsTouched() înainte de if (form.invalid) return — altfel utilizatorul nu vede mesajele de eroare la primul click pe Submit.

CORS – configurare API Gateway

Browserul blochează requesturile cross-origin dacă server-ul nu trimite header-ul Access-Control-Allow-Origin. În Spring Cloud Gateway, CORS se configurează în application.yml:

spring:
  cloud:
    gateway:
      default-filters:
        - DedupeResponseHeader=Access-Control-Allow-Credentials Access-Control-Allow-Origin
      globalcors:
        cors-configurations:
          '[/**]':
            allowedOrigins: "http://localhost:4200,http://localhost:4300"
            allowedMethods: "*"
            allowedHeaders: "*"
            allowCredentials: true

Port schimbat = CORS blocat: Dacă muți frontul de pe 4200 pe 4300 (sau orice alt port), trebuie să adaugi noul origin în allowedOrigins și să restartezi Gateway-ul. Fără asta, toate requesturile sunt blocate cu eroarea "No 'Access-Control-Allow-Origin' header".

Change Detection – ChangeDetectorRef

Răspunsurile HTTP vin uneori din afara zonei Angular (NgZone), iar UI-ul nu se actualizează automat. Soluție: injectează ChangeDetectorRef și apelează detectChanges() după ce modifici starea în error handler:

private cdr = inject(ChangeDetectorRef);

// În error handler:
error: err => {
  this.errorMessage = err.status === 401
    ? 'Utilizator sau parolă greșite.'
    : 'Eroare de server. Încearcă din nou.';
  this.loading = false;
  this.cdr.detectChanges(); // forțează update UI
}

Întrebări examen

1. Ce diferență există între Angular standalone components și NgModule?
NgModule grupează componente, directive și pipe-uri într-un modul care trebuie declarat și importat explicit. Standalone components importă direct dependențele lor — nu necesită un modul intermediar. Angular 17+ recomandă abordarea standalone ca default.
2. De ce se folosește localStorage pentru "ține-mă minte" și sessionStorage altfel?
localStorage persistă datele după închiderea browserului (până la ștergere explicită). sessionStorage se șterge automat când tab-ul sau browserul e închis. Pentru "ține-mă minte" vrem persistență îndelungată → localStorage. Pentru sesiuni normale vrem ca token-ul să expire la închiderea browserului → sessionStorage.
3. Ce este CORS și de ce e necesar în această arhitectură?
CORS (Cross-Origin Resource Sharing) e un mecanism de securitate al browserului care blochează requesturile HTTP către un origin diferit de cel al paginii. Frontend-ul rulează pe portul 4300, backend-ul pe 8080 — origini diferite. Gateway-ul trebuie să trimită headerul Access-Control-Allow-Origin ca să permită requesturile.
4. De ce adăugăm Refresh-Token în header la logout și nu în body?
Backend-ul a fost implementat să citească refresh token-ul din header-ul Refresh-Token, nu din body. Această convenție e comună pentru token-uri de revocare — body-ul unui POST de logout e de obicei gol. Dacă nu trimiți header-ul, backend-ul nu poate identifica ce token să revoce și returnează 500.
5. Ce face markAllAsTouched() și de ce e necesar la submit?
Angular afișează mesajele de validare doar pentru câmpurile "touched" (care au primit focus și l-au pierdut). Dacă utilizatorul apasă Submit fără să atingă niciun câmp, erorile nu apar. markAllAsTouched() marchează toate câmpurile ca touched simultan, forțând afișarea tuturor mesajelor de eroare.
6. Cum funcționează un validator de grup în Angular Reactive Forms?
Un validator de grup primește întregul FormGroup ca argument și poate accesa orice câmp. Returnează un obiect de erori sau null. Se aplică la al doilea argument al fb.group(), în proprietatea validators. Erorile se citesc pe form.errors, nu pe câmpul individual.
7. De ce withFetch() poate cauza probleme cu error handling?
Fetch API și XMLHttpRequest gestionează diferit răspunsurile non-2xx. Cu Fetch, unele răspunsuri 4xx apar ca ERR_ABORTED în Chrome și nu propagă corect eroarea prin pipeline-ul RxJS al Angular HttpClient. Rezultatul: error handler-ul din componentă nu se execută, UI-ul rămâne în starea "loading". Soluție: omite withFetch() și folosește XHR standard.
8. Ce este ChangeDetectorRef și când e necesar?
ChangeDetectorRef permite declanșarea manuală a detecției de schimbări Angular. E necesar când starea componentei se modifică în afara zonei Angular (NgZone) — de exemplu, în callback-uri asincrone venite din biblioteci externe sau din anumite scenarii de HTTP error handling. detectChanges() forțează re-renderizarea template-ului imediat.

Pasul 2 — Configurează app.config.ts

Înlocuiește conținutul src/app/app.config.ts:

import { ApplicationConfig } from '@angular/core';
import { provideRouter } from '@angular/router';
import { provideHttpClient, withInterceptors } from '@angular/common/http';
import { routes } from './app.routes';
import { jwtInterceptor } from './interceptors/jwt.interceptor';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(routes),
    provideHttpClient(
      withInterceptors([jwtInterceptor])

Pasul 3 — Creează AuthService 
ng generate service services/auth

Editează src/app/services/auth.service.ts:

@Injectable({ providedIn: 'root' })
export class AuthService {
  private apiUrl = environment.apiUrl;

  constructor(private http: HttpClient) {}

  login(username: string, password: string, rememberMe: boolean) {
    return this.http.post<{accessToken: string, refreshToken: string}>(
      `${this.apiUrl}/auth/login`,
      { username, password, rememberMe }
    ).pipe(
      tap(response => {
        // rememberMe → localStorage (persistent); altfel → sessionStorage (temporar)
        const storage = rememberMe ? localStorage : sessionStorage;
        storage.setItem('accessToken', response.accessToken);
        storage.setItem('refreshToken', response.refreshToken);
      })
    );
  }

  getToken(): string | null {
    return localStorage.getItem('accessToken') || sessionStorage.getItem('accessToken');
  }

  isLoggedIn(): boolean {
    return !!this.getToken();
  }

  logout() {
    localStorage.removeItem('accessToken');
    sessionStorage.removeItem('accessToken');
  }
}
Pasul 4 — Creează Login Component cu Reactive Forms 
ng generate component features/auth/login --standalone

În login.ts:

@Component({ standalone: true, imports: [ReactiveFormsModule, RouterLink], ... })
export class LoginComponent {
  form = new FormGroup({
    username: new FormControl('', [Validators.required]),
    password: new FormControl('', [Validators.required, Validators.minLength(6)]),
    rememberMe: new FormControl(false)
  });

  constructor(private authService: AuthService, private router: Router) {}

  onSubmit() {
    if (this.form.invalid) return;
    const { username, password, rememberMe } = this.form.value;
    this.authService.login(username!, password!, rememberMe!).subscribe({
      next: () => this.router.navigate(['/dashboard']),
      error: () => alert('Credențiale incorecte')
    });
  }
}
