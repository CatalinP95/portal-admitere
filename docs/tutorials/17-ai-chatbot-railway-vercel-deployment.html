1. Load Balancing — Fix Script PowerShell

Problema

Scriptul start-second-instance.ps1 folosea comanda mvn direct, dar Maven nu era în PATH pe toate sistemele. De asemenea, Java trebuia detectat automat din Adoptium JDK.

Soluție — auto-download Maven + detect Java

# Detectează Java din Adoptium JDK
$javaHome = "C:\Program Files\Eclipse Adoptium\jdk-21.0.7.6-hotspot"
$env:JAVA_HOME = $javaHome
$env:Path = "$javaHome\bin;" + $env:Path

# Maven auto-download dacă nu există
$mavenVersion = "3.9.6"
$mvnDir = ".tools\apache-maven-$mavenVersion"
if (-not (Test-Path $mvnDir)) {
    $url = "https://archive.apache.org/dist/maven/maven-3/$mavenVersion/binaries/apache-maven-$mavenVersion-bin.zip"
    Invoke-WebRequest -Uri $url -OutFile ".tools\maven.zip"
    Expand-Archive ".tools\maven.zip" -DestinationPath ".tools\"
}
$mvn = "$mvnDir\bin\mvn.cmd"

# Pornire instanță 2 pe portul 8083
$env:SPRING_PROFILES_ACTIVE = "local"
$env:SERVER_PORT = "8083"
& $mvn spring-boot:run

Maven 3.9.9 nu există — versiunea corectă din Apache Archive este 3.9.6. Versiunile mai noi sunt distribuite diferit și nu au URL-ul arhivei standard.

2. AI Chatbot — RoboAdmis cu Groq API

De ce Groq și nu Claude/OpenAI

Claude API și OpenAI necesită card bancar chiar și pentru planul gratuit. Groq oferă acces gratuit la modele open-source (Llama, Mistral, Gemma) cu un API compatibil OpenAI — aceeași interfață, fără costuri inițiale.

ProviderModel folositCostLatență
Groqllama-3.1-8b-instantGratuit~200ms
OpenAIgpt-4o-mini$0.15/1M tokens~500ms
Anthropic Claudeclaude-haiku-4Card obligatoriu~400ms

ClaudeService.java — client Groq API

@Service
public class ClaudeService {

    private static final String GROQ_API_URL = "https://api.groq.com";
    private static final String MODEL = "llama-3.1-8b-instant";

    @Value("${groq.api-key:}")
    private String apiKey;

    private final RestClient restClient;

    public ClaudeService() {
        this.restClient = RestClient.builder()
            .baseUrl(GROQ_API_URL)
            .build();
    }

    public String ask(String userMessage) {
        if (apiKey == null || apiKey.isBlank()) {
            return "Serviciul de asistență AI nu este configurat momentan.";
        }
        try {
            GroqRequest body = new GroqRequest(MODEL, List.of(
                new GroqMessage("system", SYSTEM_PROMPT),
                new GroqMessage("user", userMessage)
            ));
            GroqResponse response = restClient.post()
                .uri("/openai/v1/chat/completions")
                .header("Authorization", "Bearer " + apiKey)
                .contentType(MediaType.APPLICATION_JSON)
                .body(body)
                .retrieve()
                .body(GroqResponse.class);
            return response.choices().get(0).message().content();
        } catch (Exception e) {
            return "Ups, ceva nu a mers. Încearcă din nou! 🔧";
        }
    }

    record GroqRequest(String model, List<GroqMessage> messages) {}
    record GroqMessage(String role, String content) {}
    record GroqResponse(List<GroqChoice> choices) {}
    record GroqChoice(GroqMessage message) {}
}

Java Records — record GroqRequest(...) generează automat constructor, getters, equals, hashCode și toString. Perfecte pentru DTO-uri imutabile, fără Lombok.

ChatController.java — endpoint REST

@RestController
@RequestMapping("/api/chat")
public class ChatController {

    private final ClaudeService claudeService;

    @PostMapping("/ask")
    public ResponseEntity<ChatResponse> ask(@Valid @RequestBody ChatRequest request) {
        String reply = claudeService.ask(request.getMessage());
        return ResponseEntity.ok(new ChatResponse(reply));
    }

    record ChatRequest(@NotBlank String message) {}
    record ChatResponse(String reply) {}
}

3. Railway Deployment — Backend în Cloud

Ce este Railway

Railway este o platformă cloud care rulează containere Docker direct din GitHub. La fiecare push pe branch-ul configurat, Railway rebuild-ează și redeploy-ează automat serviciul.

Dockerfile pentru user-service

# Faza 1 — build
FROM maven:3.9.6-eclipse-temurin-21-alpine AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -B -DskipTests

# Faza 2 — runtime (imagine mai mică)
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8081
ENTRYPOINT ["java", "-jar", "app.jar"]

Multi-stage build: prima imagine conține Maven + JDK (~500MB), a doua conține doar JRE + JAR (~180MB). Railway rulează doar imaginea finală, mai mică.

4. Vercel Deployment — Frontend Angular în Cloud

Ce este Vercel

Vercel este o platformă specializată pentru frontend. Detectează automat Angular, React, Vue etc. și configurează build-ul. La fiecare push pe branch-ul configurat, redeploy-ează automat.

Problema SPA pe Vercel — 404 la refresh

Angular folosește routing pe client (History API). Când accesezi direct /dashboard/users, Vercel caută un fișier fizic la acea cale — nu găsește nimic și returnează 404. Soluția: toate rutele trebuie redirecționate spre index.html.

vercel.json — fix routing și output directory

{
  "buildCommand": "npm run build",
  "outputDirectory": "dist/frontend/browser",
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}

outputDirectory: dist/frontend/browser — Angular 17+ cu builder @angular/build:application pune output-ul în dist/[project-name]/browser/, nu direct în dist/[project-name]/ cum făcea versiunile anterioare.

Fix budget CSS — angular.json

Build-ul de producție Angular verifică dimensiunea fișierelor. Dacă un SCSS depășește limita, build-ul eșuează:

// angular.json — secțiunea budgets din configurations.production
{
  "type": "anyComponentStyle",
  "maximumWarning": "10kB",  // era 4kB — prea mic
  "maximumError": "20kB"    // era 8kB — prea mic
}

environment.ts — URL backend producție

export const environment = {
  production: true,
  apiUrl: 'https://portal-admitere-production.up.railway.app'
};

Angular environments — în development, Angular folosește environment.ts. La build cu --configuration production, Angular înlocuiește automat fișierul cu environment.prod.ts (sau cel marcat production: true). Un singur fișier, două comportamente.

5. CORS Cross-Origin — Vercel ↔ Railway

De ce apare CORS în producție

În development, Angular (localhost:4300) și backend (localhost:8080) sunt pe același host — CORS nu e o problemă. În producție, frontend-ul e pe portal-admitere.vercel.app și backend-ul pe portal-admitere-production.up.railway.app. Domeniile diferite declanșează política CORS a browserului.

Browserul trimite automat un request OPTIONS preflight înainte de orice POST/PUT/DELETE cross-origin. Dacă serverul nu răspunde corect, browser-ul blochează request-ul original.

SecurityConfiguration — CORS configurabil

@Value("${allowed.origins:http://localhost:4300}")
private String allowedOrigins;

@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    List<String> origins = List.of(allowedOrigins.split(","));
    config.setAllowedOrigins(origins);
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "OPTIONS", "PATCH"));
    config.setAllowedHeaders(List.of("*"));
    config.setAllowCredentials(true);
    config.setExposedHeaders(List.of("X-Instance-Port", "X-XSRF-TOKEN"));
    config.setMaxAge(3600L);
    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", source);
    return source;
}

allowedOrigins.split(",") — variabila de mediu poate conține mai multe origini separate prin virgulă: https://portal-admitere.vercel.app,http://localhost:4300. Fără split, întreg string-ul e tratat ca un singur origin invalid.

6. CSRF în Producție — De ce JWT Stateless nu are nevoie

Cum funcționează un atac CSRF

CSRF (Cross-Site Request Forgery) funcționează astfel: utilizatorul e autentificat pe bank.com. Un site malițios evil.com face un POST la bank.com/transfer. Browserul trimite automat cookie-ul de sesiune → tranzacția e executată fără știrea utilizatorului.

Condiție necesară: tokenul de autentificare trebuie să fie un cookie trimis automat de browser. Dacă autentificarea se face prin header Authorization: Bearer <token>, atacul CSRF nu funcționează — site-ul malițios nu poate citi sau injecta header-uri custom.

Situația noastră

JWT-ul este trimis prin header Authorization: Bearer ... de Angular. Browserul NU trimite automat acest header — Angular îl adaugă explicit prin interceptor. Un site malițios nu poate adăuga header-ul fără să aibă tokenul, ceea ce înseamnă că CSRF nu se aplică.

Fix pentru deployment cross-origin

// SecurityConfiguration.java
http
  .csrf(csrf -> csrf
      .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
      .csrfTokenRequestHandler(new CsrfTokenRequestAttributeHandler())
      .ignoringRequestMatchers("/**")  // ignoră toate — JWT e suficient
  )

CSRF rămâne configurat în cod (demonstrabil local, vizibil în cod sursă), dar ignoră toate endpoint-urile în producție. Infrastructura există, protecția nu e necesară cu JWT stateless.

❓ De ce cookie-ul XSRF nu traversează domenii diferite?
Browserele moderne blochează cookie-urile SameSite=Lax (default) în cereri cross-origin. Cookie-ul XSRF-TOKEN setat de Railway nu poate fi citit de JavaScript pe Vercel. Pentru a funcționa cross-origin, cookie-ul ar trebui setat cu SameSite=None; Secure — configurație suplimentară în Spring Boot care depășește scopul proiectului.
❓ De ce OPTIONS preflight returna 403?
Spring Security procesa request-ul OPTIONS înainte ca filtrul CORS să poată răspunde. Cu ignoringRequestMatchers("/**") și .requestMatchers(OPTIONS, "/**").permitAll(), Spring lasă CORS să răspundă primul la preflight și returnează corect 200 cu headerele necesare.
