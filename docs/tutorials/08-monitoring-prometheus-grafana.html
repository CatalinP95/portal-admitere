📊 Monitoring cu Prometheus & Grafana

Metrici din microservicii Spring Boot, vizualizate în timp real

Arhitectura de monitoring

Spring Boot
/actuator/prometheus
← scrape la 15s
Prometheus
:9090
← query PromQL
Grafana
:3000

Prometheus "trage" (pull model) metrici de la fiecare serviciu la intervalul de scrape. Grafana îl interogă pe Prometheus și afișează grafice.

Setup Spring Boot – Actuator + Micrometer

Dependințe în pom.xml

<!-- Actuator – expune endpointuri de monitorizare -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>

<!-- Micrometer Prometheus – format metrics compatibil Prometheus -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>

application.yml – expunere endpoint-uri

management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus,refresh
  endpoint:
    health:
      show-details: always
  metrics:
    export:
      prometheus:
        enabled: true

Verificare endpoint /actuator/prometheus

Accesează http://localhost:8081/actuator/prometheus în browser. Vei vedea mii de linii în format text:

# HELP jvm_memory_used_bytes The amount of used memory
# TYPE jvm_memory_used_bytes gauge
jvm_memory_used_bytes{area="heap",id="G1 Eden Space"} 5.24288E7
jvm_memory_used_bytes{area="heap",id="G1 Old Gen"} 1.5728640E7

# HELP http_server_requests_seconds Duration of HTTP server request handling
# TYPE http_server_requests_seconds summary
http_server_requests_seconds_count{method="GET",uri="/api/users"} 42
http_server_requests_seconds_sum{method="GET",uri="/api/users"} 1.23

prometheus.yml – configurația de scraping

global:
  scrape_interval: 15s     # la fiecare 15 secunde
  evaluation_interval: 15s

scrape_configs:
  - job_name: 'user-service'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['host.docker.internal:8081']
                   # ↑ din container Prometheus, mașina gazdă = host.docker.internal

  - job_name: 'admissions-service'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['host.docker.internal:8082']

  - job_name: 'api-gateway'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['host.docker.internal:8080']

Metrici disponibile automat

http_server_requests_seconds_count
Numărul de request-uri HTTP procesate, grupate pe method, uri, status
jvm_memory_used_bytes
Memorie JVM utilizată (heap, non-heap, per generație G1)
process_cpu_usage
Procentaj CPU utilizat de procesul Java
hikaricp_connections_active
Conexiuni active în connection pool-ul HikariCP (MySQL)
spring_rabbitmq_listener_seconds_count
Mesaje RabbitMQ procesate

Setup Grafana – conectare la Prometheus

Accesează Grafana

Deschide http://localhost:3000. Username: admin, Password: admin

Adaugă Prometheus ca datasource

Meniu stânga → Connections → Data Sources → Add data source → Prometheus → URL: http://prometheus:9090

Importă dashboard predefinit

Dashboards → Import → ID 4701 (JVM Micrometer) → selectează datasource Prometheus → Import

Vizualizează metrici live

Dashboard-ul arată heap memory, CPU, request rate, GC activity – totul în timp real

Query PromQL – exemple

QueryCe arată
rate(http_server_requests_seconds_count[5m])Request-uri pe secundă în ultimele 5 minute
jvm_memory_used_bytes{area="heap"}Memorie heap utilizată curent

Dashboard JVM recomandat: Importă Grafana dashboard ID 4701 – "JVM (Micrometer)" – pentru o vizualizare completă a memoriei, GC, threads și HTTP metrics dintr-un Spring Boot.

De ce /actuator/health dezactivează unele checks? În application.yml am dezactivat health.mongo.enabled: false și health.rabbit.enabled: false. Asta previne ca aplicația să apară "DOWN" dacă MongoDB sau RabbitMQ nu sunt pornite temporar în dev.

Întrebări frecvente la examen

Î: Ce este Micrometer și de ce e necesar pe lângă Spring Boot Actuator?
Spring Boot Actuator expune endpoint-uri de management (/actuator/health, /actuator/metrics). Micrometer este un „facade" de metrici (ca SLF4J pentru logging) — colectează metricile din aplicație și le poate exporta în multiple formate. Dependința micrometer-registry-prometheus traduce metricile Micrometer în formatul text specific Prometheus (linii de tipul metric_name{label="value"} 123.4), expus la /actuator/prometheus.
Î: Care este modelul pull vs push în monitoring și ce folosește Prometheus?
Prometheus folosește modelul pull (scraping): Prometheus se duce activ la fiecare serviciu la intervalul de scrape (15s) și descarcă metricile. Alternativa este modelul push (ex: StatsD, InfluxDB): serviciile trimit activ metricile. Avantajul pull: Prometheus controlează rata de colectare, serviciile nu trebuie să știe de Prometheus, și detectezi automat serviciile căzute (nu mai returnează metrici).
Î: Ce metrici expune automat Spring Boot fără configurare suplimentară?
HTTP: http_server_requests_seconds_count/sum — număr și durată request-uri per endpoint. JVM: jvm_memory_used_bytes, jvm_gc_pause_seconds, jvm_threads_live_threads. CPU: process_cpu_usage. Baza de date: hikaricp_connections_active/idle/pending (pool HikariCP). Cache: cache_gets_total (hits/misses Redis). Toate sunt înregistrate automat de Spring Boot Actuator + Micrometer.
Î: Ce este PromQL și dă un exemplu de query util?
PromQL (Prometheus Query Language) este limbajul de interogare al Prometheus. Exemple: rate(http_server_requests_seconds_count[5m]) — request-uri pe secundă în ultimele 5 minute (funcția rate calculează rata per secundă dintr-un counter). jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} * 100 — procentaj heap utilizat. histogram_quantile(0.95, rate(http_server_requests_seconds_bucket[5m])) — percentila 95 a timpilor de răspuns.
Î: Cum configurezi Grafana să se conecteze la Prometheus?
1. Accesezi http://localhost:3000 (admin/admin). 2. Meniu stânga → Connections → Data Sources → Add data source → Prometheus. 3. URL: http://prometheus:9090 (din interiorul Docker, Grafana ajunge la Prometheus prin numele serviciului, nu localhost). 4. Save & Test. Importul unui dashboard predefinit (ID 4701 = JVM Micrometer) oferă imediat grafice pentru heap, CPU, GC, threads, HTTP.
Î: De ce dezactivăm health checks pentru MongoDB și RabbitMQ în dev?
Spring Boot Actuator verifică automat sănătatea tuturor dependințelor configurate. Dacă MongoDB sau RabbitMQ nu sunt pornite, /actuator/health returnează DOWN, chiar dacă restul aplicației funcționează perfect. În dev, nu pornim întotdeauna toate serviciile. Setând health.mongo.enabled: false și health.rabbit.enabled: false, serviciul apare UP chiar fără aceste dependințe.
Î: Ce conține fișierul prometheus.yml și ce rol joacă scrape_interval?
prometheus.yml configurează ce servicii să scrape-uiască Prometheus. Fiecare job are un job_name, metrics_path (de obicei /actuator/prometheus) și targets (adrese host:port). scrape_interval: 15s = Prometheus colectează metrici la fiecare 15 secunde. Valori mai mici = rezoluție mai bună dar mai multă presiune pe servicii. În producție, 15-30s e standard. Fișierul este montat în containerul Prometheus via volumes: ./prometheus.yml:/etc/prometheus/prometheus.yml.

Implementare pas cu pas — Monitoring cu Prometheus și Grafana

Prerequisite: Docker Compose cu Prometheus și Grafana pornit (Tutorial 05). Spring Boot application rulând pe port 8081.

Pasul 1 — Adaugă dependințele Actuator și Micrometer

În user-service/pom.xml:

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
Pasul 2 — Configurează Actuator în application.yml 
management:
  endpoints:
    web:
      exposure:
        include: health, info, prometheus, refresh
        # "prometheus" expune metricile în format Prometheus la /actuator/prometheus
  endpoint:
    health:
      show-details: always
  metrics:
    export:
      prometheus:
        enabled: true
Pasul 3 — Configurează prometheus.yml

Creează prometheus.yml în rădăcina proiectului (montat în containerul Prometheus):

global:
  scrape_interval: 15s    # colectează metrici la fiecare 15 secunde

scrape_configs:
  - job_name: 'user-service'
    metrics_path: '/actuator/prometheus'
    static_configs:
      - targets: ['host.docker.internal:8081']
        # host.docker.internal = IP-ul calculatorului gazdă, vizibil din container Docker

Asigură-te că în docker-compose.yml fișierul e montat:

prometheus:
  volumes:
    - ./prometheus.yml:/etc/prometheus/prometheus.yml
Pasul 4 — Importă dashboard-ul în Grafana

Deschide http://localhost:3000 (admin/admin) și urmează pașii:

  1. Left sidebar → Connections → Data sources → Add data source
  2. Selectează Prometheus, URL: http://prometheus:9090 → Save & Test
  3. Left sidebar → Dashboards → Import
  4. Introdu ID-ul 19004 (Spring Boot Statistics) → Load → selectează Prometheus → Import

Sau dacă ai dashboard JSON propriu, plasează-l în grafana/dashboards/spring-boot.json și configurează provisioning-ul.

Verificare:

# 1. Metrici expuse de user-service:
http://localhost:8081/actuator/prometheus
# → pagină lungă cu sute de metrici în format text

# 2. Prometheus primește date:
http://localhost:9090/targets
# → user-service trebuie să apară cu State: UP

# 3. Grafana vizualizează:
http://localhost:3000
# → dashboard-ul Spring Boot afișează grafice pentru CPU, memorie, HTTP requests
