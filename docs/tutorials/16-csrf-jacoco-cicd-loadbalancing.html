1. CSRF — Cross-Site Request Forgery Protection

De ce e necesar

Fără CSRF, un site malițios poate determina browserul utilizatorului autentificat să facă cereri neautorizate (POST/PUT/DELETE) către aplicația noastră. Chiar dacă folosim JWT, browserul trimite automat cookie-urile — de aceea avem nevoie de un token suplimentar pe care atacatorul nu îl poate citi.

Notă arhitecturală: CSRF este relevant când tokenul de autentificare e stocat în cookie. Dacă JWT-ul e trimis doar prin header Authorization: Bearer, CSRF nu e necesar. Proiectul nostru demonstrează ambele mecanisme.

Flux CSRF complet

Angular pornește
→
GET /auth/csrf
→
Spring setează cookie XSRF-TOKEN
→
Angular citește cookie
→
POST cu X-XSRF-TOKEN header
→
Spring validează → 200 OK

Backend — SecurityConfiguration.java

@Bean
public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf
            .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
            .csrfTokenRequestHandler(new CsrfTokenRequestAttributeHandler())
            .ignoringRequestMatchers(
                "/auth/login", "/auth/register",
                "/auth/refresh", "/auth/logout",
                "/actuator/**", "/api/health/**"
            )
        )
        // ... rest of config
}

CookieCsrfTokenRepository.withHttpOnlyFalse() — cookieul XSRF-TOKEN nu are HttpOnly, deci Angular JavaScript îl poate citi.

CsrfTokenRequestAttributeHandler — Spring Security 6, citește tokenul din header sau parametru.

ignoringRequestMatchers — endpoint-urile de autentificare sunt excluse (nu există token CSRF la primul login).

Endpoint inițializare cookie — AuthController.java

@GetMapping("/auth/csrf")
public ResponseEntity<Void> csrf(HttpServletRequest request, HttpServletResponse response) {
    CsrfToken token = (CsrfToken) request.getAttribute(CsrfToken.class.getName());
    if (token != null) token.getToken(); // forțează scrierea cookie-ului
    return ResponseEntity.ok().build();
}

Spring Security 6 evaluează token-ul lazy — cookieul nu se scrie decât dacă cineva apelează explicit getToken(). Endpoint-ul /auth/csrf forțează această inițializare.

Frontend — app.config.ts

export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(
      withInterceptors([jwtInterceptor]),
      withXsrfConfiguration({
        cookieName: 'XSRF-TOKEN',
        headerName: 'X-XSRF-TOKEN'
      })
    ),
    provideAppInitializer(async () => {
      const http = inject(HttpClient);
      await firstValueFrom(
        http.get(`${environment.apiUrl}/auth/csrf`, { withCredentials: true })
      ).catch(() => {});
    })
  ]
};

withXsrfConfiguration — Angular citește automat cookie-ul și trimite valoarea ca header la fiecare POST/PUT/DELETE.

provideAppInitializer — se execută o singură dată la pornirea aplicației, înainte de orice altă navigare.

Fix teste MockMvc

Cu CSRF activat, toate testele de tip POST/PUT/DELETE trebuie să includă .with(csrf()):

import static org.springframework.security.test.web.servlet.request.SecurityMockMvcRequestPostProcessors.csrf;

mockMvc.perform(post("/api/users").with(csrf())
        .header("Authorization", "Bearer " + adminToken)
        .contentType(MediaType.APPLICATION_JSON)
        .content(objectMapper.writeValueAsString(req)))
    .andExpect(status().isOk());
❓ De ce login-ul nu are nevoie de CSRF?
La primul login nu există token CSRF (cookieul nu a fost setat încă). De aceea /auth/login e în ignoringRequestMatchers. Endpoint-ul /auth/csrf inițializează cookieul înainte de login.

2. JaCoCo — Acoperire cod 81%

Ce măsoară JaCoCo

JaCoCo (Java Code Coverage) instrumentează bytecode-ul Java și înregistrează ce linii, ramuri și instrucțiuni au fost executate în timpul testelor. Rezultatul este un raport HTML detaliat.

MetricValoareSemnificație
Instructions81%Instrucțiuni bytecode executate
Branches~70%Ramuri if/else/switch acoperite
Lines~83%Linii de cod executate
Methods~88%Metode apelate cel puțin o dată

Configurare pom.xml

<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <executions>
        <execution>
            <id>prepare-agent</id>
            <goals><goal>prepare-agent</goal></goals>
        </execution>
        <execution>
            <id>report</id>
            <phase>test</phase>
            <goals><goal>report</goal></goals>
        </execution>
    </executions>
</plugin>

Servire raport în aplicație

@Configuration
public class CoverageResourceConfiguration implements WebMvcConfigurer {
    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        String userDir = System.getProperty("user.dir");
        Path direct = Paths.get(userDir, "target", "site", "jacoco");
        Path withSub = Paths.get(userDir, "user-service", "target", "site", "jacoco");
        Path jacocoDir = Files.exists(direct) ? direct : withSub;
        registry.addResourceHandler("/coverage/**")
                .addResourceLocations("file:" + jacocoDir.toString().replace("\\", "/") + "/");
    }
}

De ce detectăm dinamic calea? user.dir (working directory la pornire) diferă în funcție de cum pornești aplicația. Dacă pornești din root-ul proiectului, calea include user-service/. Dacă pornești din directorul serviciului, nu.

3. Load Balancing demonstrabil

Conceptul

Spring Cloud LoadBalancer distribuie cererile între mai multe instanțe ale aceluiași serviciu folosind algoritmul Round Robin. Fiecare cerere ajunge la o instanță diferită.

Angular (4300)
→
API Gateway (8080)
→
lb://user-service
→
Instanța 1: 8081
sau
Instanța 2: 8083

InstanceHeaderFilter.java

@Component
@Order(1)
public class InstanceHeaderFilter extends OncePerRequestFilter {

    @Value("${server.port}")
    private int port;

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res,
                                      FilterChain chain) throws ServletException, IOException {
        res.setHeader("X-Instance-Port", String.valueOf(port));
        chain.doFilter(req, res);
    }
}

Pornire a doua instanță (start-second-instance.ps1)

cd user-service
mvn spring-boot:run "-Dspring-boot.run.arguments=--server.port=8083"

Demonstrare

1. Pornești ambele instanțe (8081 + 8083) → amândouă apar în Eureka la http://localhost:8761

2. F12 → Network → orice request → Response Headers → X-Instance-Port: 8081

3. La refresh → X-Instance-Port: 8083 → Round Robin confirmat vizual

4. Grafana Auto-provisioning

Problema fără provisioning

Fără configurare automată, la fiecare docker compose up Grafana pornește fără datasource și fără dashboard-uri — trebuie adăugate manual. Cu provisioning, totul e gata la primul pornit.

Structura fișierelor

grafana/
├── provisioning/
│   ├── datasources/
│   │   └── prometheus.yml    ← conectare la Prometheus
│   └── dashboards/
│       └── default.yml       ← unde caută Grafana dashboard-uri JSON
└── dashboards/
    └── spring-boot.json      ← dashboard-ul propriu-zis

prometheus.yml (datasource)

apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    url: http://prometheus:9090
    isDefault: true
    editable: false

default.yml (provider dashboard-uri)

apiVersion: 1
providers:
  - name: Default
    type: file
    options:
      path: /var/lib/grafana/dashboards

docker-compose.yml (montare volume)

grafana:
  image: grafana/grafana:latest
  volumes:
    - ./grafana/provisioning:/etc/grafana/provisioning
    - ./grafana/dashboards:/var/lib/grafana/dashboards

Dacă dashboard-ul nu apare după modificarea volumelor: containerul a fost creat înainte de adăugarea volumelor. Soluție: docker compose up -d --force-recreate grafana

5. CI/CD — GitHub Actions

Conceptul

La fiecare push pe master care modifică fișiere din user-service/, GitHub Actions rulează automat testele și construiește JAR-ul.

.github/workflows/ci.yml

name: CI — User Service

on:
  push:
    branches: [ master ]
    paths:
      - 'user-service/**'
      - '.github/workflows/ci.yml'

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
          cache: maven

      - name: Rulare teste
        working-directory: user-service
        run: mvn test -B

      - name: Build JAR
        working-directory: user-service
        run: mvn package -B -DskipTests

      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: jacoco-report
          path: user-service/target/site/jacoco/
          retention-days: 7
ElementRol
paths:Rulează DOAR dacă există modificări în user-service (nu la push pe alte servicii)
cache: mavenCacheează dependențele Maven între rulări → mai rapid
if: always()Uploadează raportul JaCoCo chiar dacă testele eșuează
temurinDistribuție OpenJDK gratuită de la Eclipse Adoptium

6. Remember Me — JWT cu TTL extins

Implementare backend

private static final long REMEMBER_ME_ACCESS_MS  = 7L  * 24 * 3600 * 1000;  // 7 zile
private static final long REMEMBER_ME_REFRESH_MS = 30L * 24 * 3600 * 1000; // 30 zile

public AuthResponse login(LoginRequest request) {
    // autentificare...
    if (request.isRememberMe()) {
        saveRefreshToken(user, rawRefreshToken, REMEMBER_ME_REFRESH_MS);
    } else {
        saveRefreshToken(user, rawRefreshToken, refreshExpirationMs); // 7 zile default
    }
}

Implementare frontend

// Cu bifă Remember Me → token persistent după închiderea browserului
if (rememberMe) {
  localStorage.setItem('accessToken', response.accessToken);
} else {
  // Fără bifă → token șters la închiderea tab-ului
  sessionStorage.setItem('accessToken', response.accessToken);
}
❓ Cum verifici în browser?
F12 → Application → Storage. Local Storage: token persistent. Session Storage: token temporar. Încearcă să închizi și să redeschizi tab-ul — cu Remember Me bifat, rămâi autentificat.

7. Pagina Coverage integrată în Angular

Problema X-Frame-Options

Spring Security adaugă implicit headerul X-Frame-Options: DENY care blochează afișarea paginilor în <iframe> din alte origini. Browserul refuză să încarce conținutul.

Soluția — filtrare header prin API Gateway

# api-gateway/application.yml
- id: coverage-report
  uri: lb://user-service
  predicates:
    - Path=/coverage/**
  filters:
    - RemoveResponseHeader=X-Frame-Options
    - RemoveResponseHeader=Content-Security-Policy

Cererea trece prin gateway pe portul 8080 (același cu API-ul Angular), iar gateway-ul elimină headerele problematice înainte să trimită răspunsul către browser.

Componenta Angular (coverage.ts)

@Component({ standalone: true, templateUrl: './coverage.html' })
export class CoverageComponent {
  coverageUrl: SafeResourceUrl;

  constructor(sanitizer: DomSanitizer) {
    // bypassSecurityTrustResourceUrl — permitem URL extern în iframe
    this.coverageUrl = sanitizer.bypassSecurityTrustResourceUrl(
      `${environment.apiUrl}/coverage/index.html`
    );
  }
}

Ruta (app.routes.ts)

{
  path: 'coverage',
  canActivate: [roleGuard],
  data: { roles: ['ADMIN'] },
  loadComponent: () => import('./features/dashboard/coverage/coverage')
    .then(m => m.CoverageComponent)
}

Implementare pas cu pas — CSRF, JaCoCo, CI/CD și Load Balancing

Prerequisite: user-service cu Spring Security configurat (Tutorial 09). Frontend Angular cu HttpClient (Tutorial 13).

Pasul 1 — Activează protecția CSRF în backend

În SecurityConfiguration.java, înlocuiește csrf.disable() cu:

.csrf(csrf -> csrf
    .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
    // withHttpOnlyFalse() = Angular poate citi cookie-ul XSRF-TOKEN din JavaScript
    .csrfTokenRequestHandler(new CsrfTokenRequestAttributeHandler())
    // CsrfTokenRequestAttributeHandler = Spring 6 nu mai evaluează lazy token-ul
    .ignoringRequestMatchers("/auth/login", "/auth/register",
                             "/auth/refresh", "/auth/logout",
                             "/actuator/**", "/api/health/**")
    // Endpoint-urile de login/register nu pot trimite CSRF (nu au token încă)
)

Adaugă endpoint dedicat pentru inițializarea cookie-ului în AuthController.java:

@GetMapping("/csrf")
public ResponseEntity<Void> csrf(HttpServletRequest request, HttpServletResponse response) {
    CsrfToken token = (CsrfToken) request.getAttribute(CsrfToken.class.getName());
    if (token != null) token.getToken();   // forțează scrierea cookie-ului în răspuns
    return ResponseEntity.ok().build();
}

Adaugă /auth/csrf la permitAll() și la ignoringRequestMatchers nu (trebuie protejat de CSRF pentru că îl inițializăm).

Pasul 2 — Configurează Angular să trimită token-ul CSRF

În src/app/app.config.ts, modifică provideHttpClient:

import { withXsrfConfiguration, provideHttpClient, withInterceptors } from '@angular/common/http';
import { provideAppInitializer, inject } from '@angular/core';
import { firstValueFrom } from 'rxjs';

export const appConfig: ApplicationConfig = {
  providers: [
    provideHttpClient(
      withInterceptors([jwtInterceptor]),
      withXsrfConfiguration({
        cookieName: 'XSRF-TOKEN',    // Angular citește cookie-ul cu acest nume
        headerName: 'X-XSRF-TOKEN'  // și îl trimite ca header la POST/PUT/DELETE
      })
    ),
    provideAppInitializer(async () => {
      const http = inject(HttpClient);
      // La startup, Angular face un GET la /auth/csrf care forțează crearea cookie-ului
      await firstValueFrom(
        http.get(`${environment.apiUrl}/auth/csrf`, { withCredentials: true })
      ).catch(() => {});
    })
  ]
};
