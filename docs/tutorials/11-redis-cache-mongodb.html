Redis Cache + MongoDB AuditLog

@Cacheable, @CacheEvict, @EnableCaching, @Document, @Async, performance demo

1. Ce este Redis și de ce îl folosim?

Redis (Remote Dictionary Server) este o bază de date in-memory de tip key-value. Datele sunt stocate în RAM, nu pe disc, deci accesul este extrem de rapid (microsecunde față de milisecunde la MySQL).

CaracteristicăMySQLRedis
StocareDisc (HDD/SSD)RAM (memorie)
Viteză1-100ms0.1-1ms
Structuri de dateTabele relaționaleString, Hash, List, Set, Sorted Set
PersistențăPermanentăOpțională (RDB/AOF)
Caz de utilizareDate principale ale aplicațieiCache, sesiuni, rate limiting, queues

Use case în proiect: Cachează profilurile utilizatorilor. La prima accesare a GET /api/profile/1, profilul se încarcă din MySQL și se salvează în Redis cu TTL 10 minute. Apelurile următoare servesc din Redis — fără query SQL.

2. Configurare Redis în Spring Boot

# application.yml
spring:
  data:
    redis:
      host: localhost
      port: 6379         # portul standard Redis
  cache:
    type: redis          # Spring Cache să folosească Redis
    redis:
      time-to-live: 600000  # TTL = 10 minute (ms) — după acest timp, cheia dispare automat
// UserServiceApplication.java
@SpringBootApplication
@EnableCaching   // OBLIGATORIU pentru a activa @Cacheable/@CacheEvict
public class UserServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(UserServiceApplication.class, args);
    }
}

3. @Cacheable – citire din cache

@Service
public class UserProfileService {

    @Cacheable(value = "profiles", key = "#userId")
    public UserProfileDto getByUserId(Long userId) {
        log.info("Cache MISS — loading profile from DB for user {}", userId);
        // Această metodă se execută DOAR la cache miss
        // La cache hit, Spring returnează direct din Redis fără a intra în metodă
        auditLogService.log(userId, "READ", "UserProfile", "Loaded profile for user " + userId);
        return userProfileRepository.findByUserId(userId)
                .map(UserProfileDto::from)
                .orElseThrow(() -> new IllegalArgumentException("Profile not found: " + userId));
    }
}

Cum funcționează @Cacheable pas cu pas

GET /api/profile/1
→
Spring verifică Redis: cheia "profiles::1"
→
MISS: nu există în Redis
→
Execută metoda → query MySQL
→
Salvează rezultat în Redis cu TTL 10min
→
Returnează răspuns
GET /api/profile/1 (a doua oară)
→
Spring verifică Redis: cheia "profiles::1"
→
HIT: există în Redis!
→
Returnează direct din Redis (fără MySQL)

5. Serializare în Redis – de ce avem nevoie de Serializable

// UserProfileDto.java — trebuie să implementeze Serializable
public class UserProfileDto implements Serializable {

    private static final long serialVersionUID = 1L;
    // serialVersionUID identifică versiunea clasei la deserializare
    // Dacă modifici clasa, incrementează serialVersionUID altfel
    // deserializarea eșuează cu InvalidClassException

    private Long id;
    private String firstName;
    private String lastName;
    ...
}

De ce Serializable? Spring Cache (Redis) serializează obiectul în bytes pentru stocare. Fără Serializable, obții NotSerializableException. Redis poate stoca orice obiect serializabil. Alternative: Jackson (JSON serialization) sau Kryo, dar Spring Boot implicit folosește Java serialization.

6. Performance Demo – cache vs DB

// UserProfileController.java — endpoint demo
@GetMapping("/{userId}/perf")
@PreAuthorize("hasRole('ADMIN')")
public ResponseEntity<Map<String, Object>> perfDemo(@PathVariable Long userId) {

    // Prima măsurătoare: din cache (dacă există)
    long t1 = System.currentTimeMillis();
    userProfileService.getByUserId(userId);
    long cached = System.currentTimeMillis() - t1;

    // Invalidăm cache-ul
    userProfileService.evictCache(userId);

    // A doua măsurătoare: forțat din DB (cache miss)
    long t2 = System.currentTimeMillis();
    userProfileService.getByUserId(userId);
    long fromDb = System.currentTimeMillis() - t2;

    return ResponseEntity.ok(Map.of(
        "cache_ms", cached,
        "db_ms", fromDb,
        "speedup", fromDb > 0 ? (double) fromDb / cached : "N/A"
    ));
}
// Rezultat real din proiect: cache_ms=5, db_ms=30018, speedup=6003

7. Ce este MongoDB și de ce îl folosim pentru AuditLog?

MongoDB este o bază de date NoSQL orientată pe documente (JSON-like). Documentele sunt flexibile — nu au schema fixă ca tabelele SQL.

CaracteristicăMySQLMongoDB
Model dateTabele, rânduri, coloaneColecții, documente (JSON)
SchemaFixă (ALTER TABLE)Flexibilă (orice câmpuri)
Query languageSQLMongoDB Query Language (MQL)
RelațiiJOIN între tabeleEmbedding sau referințe
Cazuri de utilizareDate structurate, tranzacțiiLogs, analytics, date variabile

De ce MongoDB pentru AuditLog?
Audit logs au câmpuri variabile (detalii diferite per acțiune), volum mare (scriere intensivă), nu necesită JOIN-uri cu alte date. MongoDB este ideal: scriere rapidă, stocare flexibilă, scalabil orizontal.

8. Implementarea AuditLog – modelul MongoDB

// AuditLog.java — document MongoDB
@Document(collection = "audit_logs")  // colecția MongoDB (echivalent tabelei SQL)
public class AuditLog {

    @Id           // câmpul _id din MongoDB (String, nu Long ca la JPA!)
    private String id;

    @Indexed      // creează index pentru căutare rapidă după userId
    private Long userId;

    private String action;     // "READ", "WRITE"
    private String resource;   // "UserProfile"
    private String details;    // "Loaded profile for user 1"
    private LocalDateTime timestamp;

    public AuditLog(Long userId, String action, String resource, String details) {
        this.userId    = userId;
        this.action    = action;
        this.resource  = resource;
        this.details   = details;
        this.timestamp = LocalDateTime.now();
    }
}

10. AuditLogService – @Async pentru performanță

@Service
public class AuditLogService {

    private final AuditLogRepository auditLogRepository;

    @Async   // Execuție asincronă — nu blochează request-ul principal!
    public void log(Long userId, String action, String resource, String details) {
        // Această metodă rulează pe un thread separat din thread pool
        // Request-ul utilizatorului nu așteaptă salvarea în MongoDB
        auditLogRepository.save(new AuditLog(userId, action, resource, details));
    }

    public List<AuditLog> getLogsForUser(Long userId) {
        return auditLogRepository.findByUserIdOrderByTimestampDesc(userId);
    }
}

De ce @Async? Salvarea în MongoDB durează câteva ms. Dacă am salva sincron, utilizatorul ar astepta mai mult pentru fiecare request. Cu @Async, request-ul răspunde imediat și salvarea în MongoDB se face pe un thread de fundal. Necesită @EnableAsync pe aplicație sau pe o clasă @Configuration.

// Activare @Async — fie pe main class fie pe o clasă @Configuration
@SpringBootApplication
@EnableCaching
@EnableAsync   // activează suportul pentru @Async
public class UserServiceApplication { ... }

12. Adnotările @Cacheable, @CacheEvict, @CachePut comparate

AdnotareComportamentCând se folosește
@CacheableVerifică cache înainte de metodă. Dacă există (HIT), returnează din cache. Dacă nu (MISS), execută metoda și salvează rezultatul în cache.Metode de citire (GET)
@CacheEvictȘterge intrarea din cache (înainte sau după metodă). Nu afectează rezultatul returnat.Metode de scriere (PUT, DELETE)
@CachePutExecută MEREU metoda și salvează rezultatul în cache. Nu verifică cache-ul înainte.Când vrei să actualizezi cache-ul fără a sări metoda

14. Întrebări posibile la examen

Q: Ce este Redis și pentru ce îl folosim în proiect?
Redis este o bază de date in-memory key-value. Datele sunt stocate în RAM (nu pe disc) deci accesul este extrem de rapid (sub 1ms față de 10-100ms pentru MySQL). În proiect îl folosim ca layer de cache pentru profilurile utilizatorilor: prima accesare face query SQL și salvează rezultatul în Redis cu TTL 10 minute. Apelurile următoare servesc direct din Redis fără SQL. Speedup demonstrat: 6000x (5ms cache vs 30000ms DB rece).
Q: Ce face @Cacheable și cum funcționează intern Spring?
@Cacheable(value="profiles", key="#userId") spune Spring să verifice cache-ul "profiles" cu cheia userId înainte de execuție. Dacă există (HIT) → returnează din Redis fără a executa metoda. Dacă nu există (MISS) → execută metoda, salvează rezultatul în Redis, returnează rezultatul. Spring implementează asta prin AOP (proxy): creează un proxy în jurul metodei care interceptează apelul și decide dacă execută metoda sau returnează din cache.
Q: De ce @CacheEvict pe metoda save() și nu @CachePut?
@CacheEvict șterge intrarea din cache. La apelul următor de getByUserId(), va fi cache MISS și va încărca datele fresh din DB (cu modificările salvate). @CachePut ar actualiza cache-ul cu rezultatul returnat de save(), dar ar forța execuția întotdeauna. @CacheEvict este mai simplu și mai sigur — garantează că datele stale sunt eliminate fără a fi nevoie să returneze exact același format ca și getByUserId().
Q: De ce avem nevoie de Serializable pe UserProfileDto?
Spring Cache (cu Redis backend) serializează obiectul în bytes folosind Java serialization pentru stocare. Orice obiect stocat în Redis trebuie să implementeze java.io.Serializable. serialVersionUID = 1L este un identifier al versiunii clasei — Spring/Java îl folosește la deserializare pentru a verifica compatibilitatea. Fără Serializable, Spring aruncă NotSerializableException la prima scriere în cache.
Q: Ce diferență este între MongoDB și MySQL? Când alegi fiecare?
MySQL este relațional: tabele cu schema fixă, suport complet pentru JOIN-uri și tranzacții ACID. MongoDB este document-oriented: colecții cu documente JSON flexibile (fără schema fixă), scalabil orizontal, ideal pentru logs și date semi-structurate. Alegi MySQL pentru date cu relații clare, integritate referențială, tranzacții complexe. Alegi MongoDB pentru logs/analytics (volum mare, scriere intensivă), date cu structură variabilă, scalabilitate orizontală.
Q: Ce face @Async și de ce e important pentru AuditLog?
@Async face ca metoda să ruleze pe un thread separat din thread pool (AsyncTaskExecutor), în loc de thread-ul curent al request-ului. Fără @Async: request-ul utilizatorului așteaptă salvarea în MongoDB înainte de a răspunde (latență adăugată). Cu @Async: request-ul răspunde imediat, salvarea în MongoDB se face în fundal. Necesită @EnableAsync pe aplicație. Dezavantaj: nu poți returna un rezultat sincron (returnezi void sau CompletableFuture).
Q: De ce mock-uim AuditLogService în teste și cum?
MongoDB nu rulează în mediul de test (este exclus din autoconfigurare în application-test.yml). Dacă am lăsa AuditLogService real, ar încerca să creeze AuditLogRepository care necesită MongoTemplate — ar arunca NoSuchBeanDefinitionException. Soluție: @MockBean AuditLogService înlocuiește bean-ul real cu un mock Mockito în contextul Spring. Metodele mock-ului nu fac nimic (void) sau returnează null/valori default.

15. Implementare pas cu pas — Redis Cache și MongoDB

Prerequisite: Redis și MongoDB pornite prin Docker Compose (Tutorial 05). Dependințele Redis și MongoDB în pom.xml.

Pasul 2 — Configurează Redis în application.yml și activează caching 
spring:
  data:
    redis:
      host: localhost
      port: 6379
    mongodb:

    type: redis
    redis:

Activează caching în clasa principală sau în orice clasă @Configuration:

@SpringBootApplication
@EnableCaching    // activează procesarea adnotărilor @Cacheable, @CacheEvict
public class UserServiceApplication { ... }
Pasul 3 — Adaugă @Cacheable pe metode din Service

Entitățile care se pun în cache trebuie să implementeze Serializable:

@Entity
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    // ... câmpuri
}

Pe service, adaugă adnotările de cache:

@Service
public class UserService {

    @Cacheable(value = "users", key = "#id")
    // La primul apel: execută metoda și salvează rezultatul în Redis cu cheia "users::123"
    // La apelurile următoare cu același id: returnează din Redis, fără query la MySQL
    public UserDto findById(Long id) {
        return UserDto.from(userRepository.findById(id)
                .orElseThrow(() -> new UserNotFoundException(id)));
    }

    @CacheEvict(value = "users", key = "#id")
    // La modificare/ștergere: invalidează cache-ul pentru acest id
    public void delete(Long id) {
        userRepository.deleteById(id);
    }
}
Pasul 4 — Creează AuditLog în MongoDB

Creează model/AuditLog.java — document MongoDB, nu entitate JPA:

@Document(collection = "audit_logs")   // salvat în MongoDB, nu MySQL
public class AuditLog {
    @Id
    private String id;               // String, nu Long (MongoDB folosește ObjectId)
    private String action;           // ex: "USER_CREATED"
    private String username;
    private LocalDateTime timestamp;
}

Creează repository/AuditLogRepository.java:

public interface AuditLogRepository extends MongoRepository<AuditLog, String> { }

Creează service/AuditLogService.java cu @Async — nu blochează thread-ul principal:

@Service
public class AuditLogService {

    @Async   // se execută pe un thread separat din pool-ul de task-uri Spring
    public void log(String action, String username) {
        AuditLog log = new AuditLog();
        log.setAction(action);
        log.setUsername(username);
        log.setTimestamp(LocalDateTime.now());
        auditLogRepository.save(log);
    }
}

Activează @Async în clasa principală: @EnableAsync.

docker exec -it portal-redis redis-cli keys "*"
# → "users::1"   (cheia din cache)
