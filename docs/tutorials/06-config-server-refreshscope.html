⚙️ Config Server & @RefreshScope

Configurație externalizată și reîncărcare dinamică fără restart de aplicație

Problema: configurație "îngropată" în aplicație

Dacă ai secretul JWT în application.yml al fiecărui microserviciu, ca să îl schimbi trebuie să restartezi toate serviciile. Într-un sistem cu 10 microservicii, asta e inacceptabil.

Soluție: Spring Cloud Config Server – un serviciu central care servește configurații pentru toate microserviciile. Schimbi în Config Server, trimiți un POST la /actuator/refresh, și microserviciul preia noile valori fără restart.

Arhitectura

config-server
(port 8888)
→ servește →
user-service
(port 8081)
→ @RefreshScope →
JwtUtil preia noile valori

Config Server – setup

1. Dependința în pom.xml

<!-- config-server/pom.xml -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-config-server</artifactId>
</dependency>

2. Activare în Application.java

@SpringBootApplication
@EnableConfigServer
public class ConfigServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(ConfigServerApplication.class, args);
    }
}

3. Fișier de configurație pentru user-service

Creat la config-server/src/main/resources/config/user-service.yml:

# config-server/src/main/resources/config/user-service.yml
jwt:
  secret: 404E635266556A586E3272357538782F413F4428472B4B6250645367566B5970
  expiration: 86400000    # 24 ore
  refresh-expiration: 604800000  # 7 zile

User Service – consumă Config Server

Dependința Spring Cloud Config Client

<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-config</artifactId>
</dependency>

application.yml – importul configurației

# user-service/src/main/resources/application.yml
spring:
  application:
    name: user-service  # Config Server caută user-service.yml
  config:
    import: "optional:configserver:http://localhost:8888"
             # ↑ optional: dacă Config Server nu e pornit, aplicația merge oricum

# Expune endpoint-ul /actuator/refresh
management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus,refresh

@RefreshScope pe JwtUtil

@RefreshScope îi spune Spring că bean-ul trebuie recreat când se apelează /actuator/refresh. Noile valori @Value sunt reinjectate.

@RefreshScope   // bean-ul se recreează la POST /actuator/refresh
@Component
public class JwtUtil {

    @Value("${jwt.secret}")
    private String secret;         // reîncărcat din Config Server

    @Value("${jwt.expiration}")
    private long expiration;       // reîncărcat din Config Server

    // ... metode de generare/validare token
}

Atenție: Fără @RefreshScope, chiar dacă apelezi /actuator/refresh, JwtUtil păstrează valorile vechi din momentul pornirii aplicației. @RefreshScope este obligatoriu pe orice bean cu @Value din Config Server.

Demo pas cu pas – @RefreshScope în acțiune

Pornește Config Server + user-service

Asigură-te că ambele rulează. Config Server pe portul 8888, user-service pe 8081.

Generează un token și verifică expiry-ul pe jwt.io

POST /api/auth/login, copiază accessToken, lipește pe jwt.io → câmpul exp = timestamp Unix al expirării.

Schimbă expiration în config-server/config/user-service.yml

Modifică expiration: 3600000 (1 oră în loc de 24 ore). Salvează fișierul.

Trimite POST la /actuator/refresh

POST http://localhost:8081/actuator/refresh cu body gol. Răspuns: lista de proprietăți actualizate.

Generează un nou token și verifică pe jwt.io

Noul token are exp cu 1 oră față de iat, în loc de 24 ore. Schimbarea a luat efect fără restart!

Răspuns de la /actuator/refresh

// POST http://localhost:8081/actuator/refresh
// Răspuns:
[
  "jwt.expiration",
  "config.client.version"
]

Întrebări frecvente la examen

Î: Ce este Spring Cloud Config Server și ce problemă rezolvă?
Config Server este un microserviciu centralizat care servește configurații (proprietăți) pentru toate celelalte microservicii. Rezolvă problema „configurației îngropate" — fără el, fiecare microserviciu are propriul application.yml și o schimbare (ex: secretul JWT) necesită restart-ul tuturor. Cu Config Server: schimbi o dată, trimiți un POST la /actuator/refresh, și serviciile preiau noile valori fără restart.
Î: Ce face @EnableConfigServer și unde se pune?
@EnableConfigServer se pune pe clasa main (ConfigServerApplication.java) împreună cu @SpringBootApplication. Activează servlet-ul care servește configurațiile. Config Server citește fișierele din src/main/resources/config/ și le servește la URL-uri de forma /user-service/default. Numele fișierului (ex: user-service.yml) trebuie să corespundă cu spring.application.name al clientului.
Î: Ce face @RefreshScope și ce se întâmplă fără el?
@RefreshScope pe JwtUtil înseamnă că Spring distruge instanța curentă și creează una nouă la POST /actuator/refresh, reinjectând valorile @Value din Config Server. Fără @RefreshScope: bean-ul este singleton standard, creat o singură dată la startup. Chiar dacă trimiți refresh, JwtUtil păstrează secreta și expiry-ul originale — schimbarea nu are efect.
Î: Ce înseamnă optional: în spring.config.import și de ce e important?
spring.config.import: "optional:configserver:http://localhost:8888" — prefix-ul optional: spune Spring că dacă Config Server nu e disponibil la pornire, aplicația merge oricum cu valorile locale din application.yml. Fără optional:, dacă Config Server nu e pornit, aplicația eșuează la startup cu o eroare de conexiune. Util în dezvoltare când nu vrei să pornești Config Server de fiecare dată.
Î: Ce returnează POST /actuator/refresh și de ce e necesar în application.yml?
Returnează un array JSON cu numele proprietăților care s-au schimbat: ["jwt.expiration", "config.client.version"]. Pentru ca endpoint-ul să fie accesibil, trebuie expus explicit în application.yml sub management.endpoints.web.exposure.include: health,info,prometheus,refresh. Implicit, Actuator nu expune refresh din motive de securitate.
Î: Cum caută Config Server configurația pentru un microserviciu?
Config Server folosește spring.application.name al clientului ca cheie de căutare. Dacă name: user-service, caută fișierul user-service.yml în directorul config/. Suportă și profiluri: user-service-dev.yml pentru profilul dev. Valorile din fișierul de profil suprascriu valorile din fișierul general, la fel ca Spring Profiles normale.
Î: De ce @RefreshScope cauzează probleme în testele JUnit?
@RefreshScope este o adnotare din Spring Cloud care înregistrează un scope custom numit refresh. Testele cu @SpringBootTest nu includ Spring Cloud în context (exclus explicit pentru a evita conexiunile la RabbitMQ/Redis). Când Spring încearcă să creeze bean-ul JwtUtil cu scope-ul refresh, aruncă No Scope registered for scope name 'refresh'. Soluția: scriem testele JwtUtil fără context Spring, folosind ReflectionTestUtils.

Implementare pas cu pas — Config Server

Prerequisite: Discovery Server (Eureka) creat și pornit. Spring Cloud BOM adăugat în pom.xml-ul părinte.

Pasul 1 — Creează modulul config-server

Generează un proiect nou la start.spring.io: dependințe Config Server + Eureka Discovery Client. Port: 8888.

Activează serverul în clasa principală:

@SpringBootApplication
@EnableConfigServer
public class ConfigServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(ConfigServerApplication.class, args);
    }
}

Configurează application.yml:

server:
  port: 8888
spring:
  application:
    name: config-server
  profiles:
    active: native          # citește din filesystem, nu din Git
  cloud:
    config:
      server:
        native:
          search-locations: classpath:/config  # folderul cu yml-urile serviciilor
eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka/
Pasul 2 — Creează fișierul de configurare pentru user-service

Creează folderul config-server/src/main/resources/config/ și în el fișierul user-service.yml. Numele fișierului = spring.application.name al clientului:

jwt:
  secret: 404E635266556A586E3272357538782F413F4428472B4B6250645367566B5970
  expiration: 86400000
  refresh-expiration: 604800000
Pasul 3 — Conectează user-service la Config Server

Adaugă dependința în user-service/pom.xml:

<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-config</artifactId>
</dependency>

În user-service/src/main/resources/application.yml:

spring:
  config:
    import: "optional:configserver:http://localhost:8888"
  application:
    name: user-service   # trebuie să coincidă cu numele fișierului user-service.yml
Pasul 4 — Adaugă @RefreshScope și expune /actuator/refresh 
@Component
@RefreshScope   // bean-ul va fi recreat la POST /actuator/refresh
public class JwtUtil {
    @Value("${jwt.secret}")
    private String secret;
    // ...
}

În application.yml al user-service:

management:
  endpoints:
    web:
      exposure:
        include: health, refresh

Verificare: Pornește config-server (8888) apoi user-service (8081).

# Verifică că user-service primește configurarea:
http://localhost:8888/user-service/default
# → JSON cu {"jwt":{"secret":"404E...","expiration":86400000}}

Pentru a testa refresh: modifică expiration în user-service.yml, apoi trimite POST http://localhost:8081/actuator/refresh — valorile se actualizează fără restart.
