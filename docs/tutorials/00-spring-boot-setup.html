Spring Boot – Structura Proiectului

@SpringBootApplication, IoC, Dependency Injection, application.yml, profile-uri

1. Ce este Spring Boot?

Spring Boot este un framework care simplifică crearea aplicațiilor Java enterprise eliminând configurările manuale. Oferă:

  • Autoconfigurare — detectează librăriile din classpath și configurează automat beans-urile necesare
  • Embedded server — rulezi cu java -jar app.jar, nu mai ai nevoie de Tomcat extern
  • Starter dependencies — spring-boot-starter-web adaugă tot ce trebuie pentru un REST API
  • Production-ready — Actuator, metrici, health checks incluse

Diferența Spring vs Spring Boot: Spring este framework-ul de bază (IoC, AOP, Security, Data). Spring Boot este un layer deasupra care elimină boilerplate-ul și configurarea XML.

2. @SpringBootApplication – punctul de intrare

Această adnotare singură înlocuiește trei adnotări separate:

@SpringBootApplication   // = @Configuration + @EnableAutoConfiguration + @ComponentScan
public class UserServiceApplication {

    public static void main(String[] args) {
        SpringApplication.run(UserServiceApplication.class, args);
    }
}
AdnotareCe face
@ConfigurationMarchează clasa ca sursă de bean definitions (metode cu @Bean)
@EnableAutoConfigurationSpring Boot scanează classpath-ul și configurează automat beans (DataSource, Security, Redis etc.)
@ComponentScanScanează pachetul curent și subpachetele după @Component, @Service, @Repository, @Controller

Important: Clasa cu @SpringBootApplication trebuie să fie în pachetul rădăcină (ex: com.campus.userservice), nu într-un subpachet, altfel @ComponentScan nu găsește celelalte clase.

4. IoC și Dependency Injection

4.1 Inversiunea Controlului (IoC)

În programarea tradițională, tu creezi obiectele: new UserService(). Cu IoC, Spring creează și gestionează obiectele (beans). Tu doar declari ce ai nevoie.

4.2 Dependency Injection (DI)

Spring injectează (pasează) dependențele automat. Există 3 moduri:

// 1. Constructor Injection — RECOMANDAT (testabil, imutable)
@Service
public class UserProfileService {
    private final UserProfileRepository userProfileRepository;
    private final UserRepository userRepository;

    public UserProfileService(UserProfileRepository repo, UserRepository userRepo) {
        this.userProfileRepository = repo;
        this.userRepository = userRepo;
    }
}

// 2. Field Injection — NU este recomandat (nu testabil ușor)
@Autowired
private UserRepository userRepository;

// 3. Setter Injection — rar folosit
@Autowired
public void setRepository(UserRepository repo) { ... }

De ce constructor injection este recomandat?
1. Câmpurile pot fi final — imutabile după construire
2. Eroarea apare la compilare, nu la runtime
3. Se poate testa ușor cu new Service(mockRepo) fără Spring

4.3 Adnotările stereotip

AdnotareUnde se puneCe face
@ComponentOrice clasăMarcheaza clasa ca bean Spring generic
@ServiceBusiness logicAlias pentru @Component; marchează semantic layer-ul de servicii
@RepositoryDAO / acces DB@Component + conversie excepții JPA în DataAccessException
@ControllerMVC controllers@Component + integrare cu Spring MVC
@RestControllerREST API controllers@Controller + @ResponseBody (JSON automat)
@ConfigurationClase de configurare@Component + sursa de @Bean definitions

4.4 @Bean – definire manuală de beans

// ApplicationConfiguration.java
@Configuration
public class ApplicationConfiguration {

    // Metoda returnează un bean gestionat de Spring
    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
        // BCrypt adaugă automat un salt random la fiecare hash
    }

    @Bean
    public AuthenticationManager authManager(AuthenticationConfiguration config) throws Exception {
        return config.getAuthenticationManager();
    }
}

eureka:

      defaultZone: http://localhost:8761/eureka  # Discovery Server

5.1 ddl-auto – ce face Hibernate cu schema DB

ValoareComportamentCând să folosești
createȘterge și recreează schema la fiecare pornireTeste, prototipuri
create-dropRecreează la start, șterge la oprireTeste (H2)
updateAdaugă coloane/tabele noi, NU ștergeDevelopment
validateVerifică că schema corespunde entităților, nu modificăProducție
noneHibernate nu face nimic cu schemaProducție cu migrații Flyway/Liquibase

6. Profile-uri Spring

Profile-urile permit configurări diferite pentru medii diferite (dev, test, prod).

# application-test.yml — activ când spring.profiles.active=test
spring:
  datasource:
    url: jdbc:h2:mem:testdb     # H2 in-memory în loc de MySQL
    driver-class-name: org.h2.Driver
  jpa:
    database-platform: org.hibernate.dialect.H2Dialect
    hibernate:
      ddl-auto: create          # recreează schema înainte de fiecare test
  autoconfigure:
    exclude:
      - org.springframework.boot.autoconfigure.amqp.RabbitAutoConfiguration
      - org.springframework.boot.autoconfigure.mongo.MongoAutoConfiguration
      - org.springframework.boot.autoconfigure.data.mongo.MongoDataAutoConfiguration
      - org.springframework.boot.autoconfigure.data.mongo.MongoRepositoriesAutoConfiguration
      - org.springframework.boot.autoconfigure.data.redis.RedisAutoConfiguration
  cache:
    type: none                  # dezactivăm Redis cache în teste
// Activare profile în test
@SpringBootTest
@ActiveProfiles("test")  // încarcă application-test.yml
class MyTest { ... }

// Activare profil la runtime
java -jar app.jar --spring.profiles.active=prod

8. Ciclul de viață al unui Bean Spring

// @PostConstruct — rulează imediat după ce bean-ul a fost injectat
@Service
public class MyService {
    @PostConstruct
    public void init() {
        System.out.println("Bean creat și injectat, inițializare...");
    }

    @PreDestroy
    public void cleanup() {
        System.out.println("Aplicația se oprește, curățenie...");
    }
}

Ordinea completă de pornire Spring Boot

  1. main() → SpringApplication.run()
  2. Se crează ApplicationContext (containerul IoC)
  3. Se încarcă application.yml și proprietăți
  4. ComponentScan — se găsesc toate clasele adnotate
  5. AutoConfiguration — Spring configurează DataSource, Security, Cache etc.
  6. Se creează beans și se injectează dependențele
  7. Se rulează metodele @PostConstruct
  8. Embedded Tomcat pornește pe portul configurat
  9. Aplicația este gata să primească requesturi

9. DTO vs Entity – de ce avem ambele?

EntityDTO (Data Transfer Object)
Mapat direct pe tabela din DBNu are legătură cu DB
Are @Entity, @Id, @ColumnClasă simplă (POJO)
Poate conține date sensibile (password hash)Conține doar ce vrem să trimitem clientului
User.javaUserDto.java — fără câmpul password
// UserDto.java — nu are password!
public class UserDto {
    private Long id;
    private String username;
    private String email;
    private String role;

    // factory method — convertire Entity → DTO
    public static UserDto from(User user) {
        UserDto dto = new UserDto();
        dto.id       = user.getId();
        dto.username = user.getUsername();
        dto.email    = user.getEmail();
        dto.role     = user.getRole().name();
        return dto;
    }
}

Pasul 4 — Creează un model simplu

Creează fișierul model/User.java:

@Entity
@Table(name = "users")
@Data                    // Lombok: generează getters/setters/toString
@NoArgsConstructor
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(unique = true, nullable = false)
    private String username;

    @Column(nullable = false)
    private String password;

    @Column(unique = true, nullable = false)
    private String email;
}
Pasul 5 — Creează Repository și Service

Creează repository/UserRepository.java:

@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByUsername(String username);
    boolean existsByEmail(String email);
}

Creează service/UserService.java:

@Service
public class UserService {

    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public List<User> findAll() {
        return userRepository.findAll();
    }
}
Pasul 6 — Creează un REST Controller

Creează api/UserController.java:

@RestController
@RequestMapping("/api/users")
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping
    public ResponseEntity<List<User>> getAll() {
        return ResponseEntity.ok(userService.findAll());
    }
}

11. Întrebări posibile la examen

Q: Ce este Spring Boot și cum diferă de Spring Framework?
Spring Boot este un layer deasupra Spring Framework care elimină configurarea manuală prin autoconfigurare. Spring Framework oferă funcționalitățile de bază (IoC, AOP, Security, Data), dar trebuie configurat explicit. Spring Boot detectează librăriile din classpath și le configurează automat. Ex: dacă ai spring-boot-starter-data-jpa și un driver MySQL, Spring Boot creează automat DataSource și EntityManager.
Q: Ce face @SpringBootApplication și din ce e compus?
@SpringBootApplication = @Configuration + @EnableAutoConfiguration + @ComponentScan. @Configuration marchează clasa ca sursă de beans. @EnableAutoConfiguration activează autoconfigurarea (detectează classpath și configurează beans). @ComponentScan scanează pachetul curent și subpachetele după clase adnotate cu @Component, @Service, @Repository, @Controller.
Q: Care este diferența dintre @Component, @Service și @Repository?
Toate sunt alias-uri pentru @Component (înregistrează clasa ca bean Spring). Diferența este semantică: @Service marchează business logic, @Repository marchează acces la date (+ conversie excepții JPA), @Controller/@RestController marchează layer-ul web. Folosirea adnotării corecte ajută la înțelegerea codului și permite Spring să aplice aspecte specifice fiecărui layer.
Q: De ce se preferă constructor injection față de field injection?
Constructor injection permite câmpurile să fie final (imutabile), erorile de dependențe circulare apar la pornire nu la runtime, și clasa poate fi instanțiată și testată fără Spring (cu new MyService(mockRepo)). Field injection cu @Autowired necesită Spring pentru a funcționa și nu poate fi testat ușor.
Q: Ce face ddl-auto: update vs validate?
update: Hibernate compară entitățile cu schema existentă și adaugă coloane/tabele noi dar NU șterge. validate: Hibernate verifică că schema existentă corespunde entităților și aruncă excepție dacă nu corespunde, dar nu modifică nimic. Pentru producție se recomandă validate sau none cu Flyway/Liquibase pentru migrații controlate.
Q: Ce este un profile Spring și când îl folosești?
Un profil permite configurări diferite pentru medii diferite. Ex: application-test.yml folosit în teste cu H2 in-memory, application-dev.yml pentru development, application-prod.yml pentru producție. Se activează cu @ActiveProfiles("test") în teste sau cu --spring.profiles.active=prod la startup. Permite dezactivarea anumitor autoconfigurări (ex: Redis, MongoDB) în teste.
Q: Ce este IoC (Inversion of Control)?
Tradițional: tu creezi obiectele de care ai nevoie (new UserService()). Cu IoC: Spring creează și gestionează obiectele (beans). Tu declari ce dependențe vrei (constructor injection) și Spring le pasează. Controlul creării obiectelor este inversat de la programator la framework. Avantaj: decuplare, testabilitate, management automat al ciclului de viață.
Q: De ce avem nevoie de DTO și nu returnăm direct Entityurile?
1. Securitate: entitățile pot conține date sensibile (password hash, intern). 2. Decuplare: clientul nu trebuie să cunoască structura internă a BD. 3. Performanță: DTO returnează doar câmpurile necesare. 4. Flexibilitate: poți schimba schema DB fără să schimbi API-ul public. Ex: UserDto nu conține câmpul password din entitatea User.
