JPA – Entități, Relații, Repository, JPQL

@Entity, @Id, @OneToMany, JpaRepository, @Transactional, query methods, JPQL

1. Ce este JPA și cum funcționează?

JPA (Java Persistence API) este un standard Java pentru maparea obiectelor Java la tabele din baza de date relațională. Hibernate este implementarea JPA folosită de Spring Boot.

  • ORM (Object-Relational Mapping) — mapează clasa Java la tabelă, câmpul la coloană, obiectul la rând
  • Hibernate — generează automat SQL din cod Java, gestionează conexiunile și tranzacțiile
  • Spring Data JPA — layer deasupra JPA care generează repository-uri automat

2. @Entity – definirea unei entități

@Entity                              // Marchează clasa ca entitate JPA (mapată pe tabelă)
@Table(name = "users")              // Specifică explicit numele tabelei (dacă diferă de clasă)
public class User {

    @Id                              // Câmpul este cheia primară (PRIMARY KEY)
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    // IDENTITY = auto-increment în MySQL (id gestionat de DB)
    // SEQUENCE = secvență (PostgreSQL)
    // AUTO = alege automat în funcție de DB
    // TABLE = tabelă separată cu contor (rar)
    private Long id;

    @Column(nullable = false, unique = true, length = 50)
    // nullable=false → NOT NULL în SQL
    // unique=true → UNIQUE constraint
    // length=50 → VARCHAR(50)
    private String username;

    @Column(nullable = false, unique = true)
    private String email;

    @Column(nullable = false)
    private String password;      // hash BCrypt, NU parola plain text

    @Enumerated(EnumType.STRING)    // stochează "STUDENT" sau "ADMIN" ca text
    // EnumType.ORDINAL ar stoca 0, 1 — PERICULOS (se poate schimba ordinea)
    @Column(nullable = false)
    private Role role = Role.STUDENT;

    @Column(nullable = false)
    private boolean enabled = true;

    // Constructori, getteri, setteri — necesari pentru JPA
    // JPA necesită un constructor fără argumente (poate fi protected)
}

3. UserProfile – @OneToOne cu User

@Entity
@Table(name = "user_profiles")
public class UserProfile {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // Owner-ul relației — UserProfile deține coloana FK user_id
    @OneToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false)
    // @JoinColumn specifică coloana FK din tabelul USER_PROFILES
    // name="user_id" → coloana se numește user_id în DB
    private User user;

    @Column(length = 100)
    private String firstName;

    @Column(length = 100)
    private String lastName;

    @Column(unique = true, length = 13)
    private String cnp;

    private LocalDate dateOfBirth;    // mapată ca DATE în SQL
    private String phone;
}

7. FetchType: LAZY vs EAGER

TipComportamentDefault pentru
LAZYDatele se încarcă doar când accesezi câmpul@OneToMany, @ManyToMany
EAGERDatele se încarcă automat cu entitatea parentă@ManyToOne, @OneToOne

Problema N+1 cu EAGER: Dacă ai 100 User-i cu EAGER pe Announcements, Hibernate face 1 query pentru useri + 100 query-uri pentru anunțurile fiecărui user = 101 query-uri! Soluție: LAZY + JOIN FETCH în JPQL când ai nevoie de date.

8. JpaRepository – toate metodele disponibile

public interface UserRepository extends JpaRepository<User, Long> {
    // JpaRepository<Entity, IdType>
    // Toate metodele de mai jos sunt disponibile automat:

    // CRUD de bază
    // save(User entity)             — INSERT sau UPDATE (merge)
    // findById(Long id)             — SELECT * WHERE id=?  → Optional<User>
    // findAll()                     — SELECT * FROM users  → List<User>
    // findAll(Pageable)             — SELECT * LIMIT x OFFSET y → Page<User>
    // deleteById(Long id)           — DELETE WHERE id=?
    // delete(User entity)           — DELETE WHERE id=entity.id
    // deleteAll()                   — DELETE FROM users (atentie!)
    // count()                       — SELECT COUNT(*)
    // existsById(Long id)           — SELECT 1 WHERE id=?

    // Metode custom derivate din nume
    Optional<User> findByUsername(String username);
    // → SELECT * FROM users WHERE username = ?

    boolean existsByUsername(String username);
    // → SELECT COUNT(*) > 0 WHERE username = ?

    boolean existsByEmail(String email);
    // → SELECT COUNT(*) > 0 WHERE email = ?

    Optional<User> findByEmail(String email);
}

10. @Query – JPQL și SQL nativ

// RefreshTokenRepository.java
public interface RefreshTokenRepository extends JpaRepository<RefreshToken, Long> {

    Optional<RefreshToken> findByToken(String token);

    // @Query cu JPQL (Java Persistence Query Language)
    // Operează pe clase Java, nu pe tabele SQL
    @Query("UPDATE RefreshToken r SET r.revoked = true WHERE r.user = :user")
    @Modifying   // obligatoriu pentru UPDATE/DELETE în @Query
    @Transactional
    void revokeAllByUser(@Param("user") User user);

    // @Query cu SQL nativ (nativeQuery=true)
    @Query(value = "SELECT * FROM refresh_tokens WHERE user_id = ?1 AND revoked = false",
           nativeQuery = true)
    List<RefreshToken> findActiveTokensForUser(Long userId);
}

// UserProfileRepository.java
public interface UserProfileRepository extends JpaRepository<UserProfile, Long> {
    Optional<UserProfile> findByUserId(Long userId);
    // → SELECT * FROM user_profiles WHERE user_id = ?

    boolean existsByCnp(String cnp);
    // → SELECT COUNT(*) > 0 WHERE cnp = ?
}

11. @Transactional – gestionarea tranzacțiilor

@Service
public class AuthService {

    @Transactional   // Toată metoda rulează într-o singură tranzacție DB
    public AuthResponse register(RegisterRequest request) {
        // Dacă oricare din aceste operații eșuează,
        // TOATE sunt anulate (rollback)
        User saved = userRepository.save(user);
        saveRefreshToken(saved, rawRefreshToken);
        // Commit automat la finalul metodei dacă nu e excepție
        return new AuthResponse(...);
    }
}
Atribut @TransactionalValoare defaultEfect
readOnlyfalsereadOnly=true optimizează query-urile de citire (Hibernate nu trackează modificările)
rollbackForRuntimeExceptionExcepțiile verificate nu fac rollback implicit — trebuie specificate
propagationREQUIREDREQUIRED: folosește tranzacția existentă sau creează una nouă
isolationDEFAULTNivelul de izolație a tranzacției (READ_COMMITTED etc.)

Regulă de bază: @Transactional se pune pe metode din @Service (business layer), nu pe @Repository sau @Controller. Pune pe operații care modifică mai multe tabele simultan și care trebuie să fie atomice.

12. JPQL – Java Persistence Query Language

JPQL operează pe clase Java, nu pe tabele SQL. Hibernate traduce JPQL în SQL.

// JPQL — folosești clasa Java și câmpul Java (nu tabelul/coloana SQL)
"SELECT u FROM User u WHERE u.role = :role"
//         ↑ clasa Java    ↑ câmpul Java

"SELECT u FROM User u WHERE u.enabled = true ORDER BY u.username ASC"

"UPDATE RefreshToken r SET r.revoked = true WHERE r.user = :user"

"SELECT p FROM UserProfile p WHERE p.user.id = :userId"
// navigare prin relații cu punct!

// JOIN FETCH — rezolvă problema N+1
"SELECT u FROM User u LEFT JOIN FETCH u.refreshTokens WHERE u.id = :id"
// Un singur query care aduce și token-urile (nu N query-uri separate)

13. Problema FK în teste și soluția

Eroare frecventă: ConstraintViolationException: user_profiles.user_id references users.id

@BeforeEach
void setUp() {
    // GREȘIT — aruncă ConstraintViolationException
    userRepository.deleteAll();     // user_profiles.user_id referențiează users.id!
    userProfileRepository.deleteAll();

    // CORECT — întâi ștergem entitățile cu FK, apoi entitatea referențiată
    userProfileRepository.deleteAll();   // are FK → users
    refreshTokenRepository.deleteAll();  // are FK → users
    userRepository.deleteAll();          // în final ștergem users
}

15. Întrebări posibile la examen

Q: Ce este JPA și care este relația cu Hibernate?
JPA (Java Persistence API) este un standard Java (specificație, nu implementare) care definește cum se mapează obiectele Java la baze de date relaționale. Hibernate este cea mai populară implementare a JPA. Spring Boot folosește Hibernate ca implementare JPA implicit. Spring Data JPA adaugă un layer deasupra care generează automat repository-urile din interfețe.
Q: Care este diferența dintre @OneToMany și @ManyToOne? Cine deține FK-ul?
@ManyToOne deține FK-ul — coloana foreign key se află în tabela entității care are @ManyToOne. @OneToMany NU deține FK-ul și trebuie să aibă mappedBy care indică câmpul din cealaltă entitate. Exemplu: RefreshToken are @ManyToOne User → tabelul refresh_tokens are coloana user_id (FK). User are @OneToMany RefreshToken mappedBy="user" → tabela users NU are o coloană extra.
Q: Ce face mappedBy și de ce e important?
mappedBy spune JPA că această parte a relației NU deține FK-ul în baza de date — cealaltă entitate îl deține. Fără mappedBy pe una din părți, JPA ar crea coloane FK pe AMBELE tabele sau o tabelă de join. Cu mappedBy="user" pe @OneToMany din User, JPA știe că FK-ul (user_id) se află în tabela RefreshToken și folosește acea coloană pentru join.
Q: Diferența dintre FetchType.LAZY și EAGER? Ce problemă rezolvă LAZY?
EAGER: Hibernate include datele asociate în query-ul principal (LEFT JOIN) — datele sunt mereu disponibile dar cresc volumul. LAZY: Hibernate generează un query separat doar când accesezi câmpul — economisești resurse dacă nu ai nevoie de date. Problema N+1: dacă ai 100 useri cu EAGER pe announcements, Hibernate face 101 query-uri (1 pentru useri + 100 pentru anunțuri). LAZY rezolvă: face query suplimentar doar când accesezi câmpul.
Q: Ce face @Transactional și când apare rollback?
@Transactional înfășoară metoda într-o tranzacție DB. Toate operațiile din metodă fie se execută complet (commit), fie niciuna (rollback). Rollback automat apare la: RuntimeException și Error. Excepțiile checked (IOException) NU fac rollback implicit — trebuie specificate cu rollbackFor. Se poate configura: @Transactional(rollbackFor = Exception.class) pentru orice excepție.
Q: Diferența dintre JPQL și SQL nativ? Când folosești fiecare?
JPQL operează pe clase Java și câmpuri Java (nu pe tabele/coloane SQL). Hibernate traduce JPQL în SQL specific fiecărei baze de date — portabil între MySQL, PostgreSQL, H2. SQL nativ (nativeQuery=true) este SQL direct, specific bazei de date — mai flexibil pentru query-uri complexe dar nu este portabil. Folosești JPQL pentru interogări standard (portabilitate), SQL nativ pentru funcții specifice DB-ului sau optimizări de performanță.
Q: Ce face @Modifying și când este necesar?
@Modifying este necesar pe metode @Query care fac UPDATE sau DELETE (nu SELECT). Fără @Modifying, Spring Data JPA aruncă InvalidDataAccessApiUsageException. Trebuie combinat cu @Transactional deoarece operațiile de modificare necesită tranzacție activă. Exemplu: revokeAllByUser() marchează toate token-urile unui user ca revocate cu un singur UPDATE.
Q: Ce este orphanRemoval și cum diferă de CascadeType.REMOVE?
CascadeType.REMOVE: când ștergi entitatea parentă, se șterg și copiii. orphanRemoval=true: când scoți un copil din colecție (parentEntity.getChildren().remove(child)) fără să îl ștergi explicit, Hibernate îl șterge din DB. orphanRemoval este mai strict — gestionează cazul când deconectezi o entitate de la parent-ul ei. De obicei se folosesc împreună: cascade=ALL, orphanRemoval=true.

16. Implementare pas cu pas — JPA Entități și Relații

Prerequisite: Spring Boot cu spring-boot-starter-data-jpa și driver MySQL în pom.xml (Tutorial 00). MySQL pornit prin Docker.

Pasul 1 — Creează entitatea principală

Creează model/User.java. Fiecare câmp devine o coloană în tabela users:

@Entity
@Table(name = "users")
@Data
@NoArgsConstructor
public class User {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // AUTO_INCREMENT în MySQL
    private Long id;

    @Column(unique = true, nullable = false, length = 50)
    private String username;

    @Column(nullable = false)
    private String password;

    @Column(unique = true, nullable = false)
    private String email;

    @Enumerated(EnumType.STRING)  // stochează "ADMIN" nu "0" în DB
    @Column(nullable = false)
    private Role role;
}

Creează enum-ul model/Role.java:

public enum Role { ADMIN, SECRETARIAT, STUDENT }
Pasul 2 — Creează relație @OneToOne

Creează model/UserProfile.java — profil detaliat legat 1:1 de User:

@Entity
@Table(name = "user_profiles")
@Data
@NoArgsConstructor
public class UserProfile {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @OneToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", unique = true)  // coloana FK în tabela user_profiles
    private User user;

    private String firstName;
    private String lastName;
    private String phone;
}

Pasul 4 — Verifică că Hibernate creează tabelele

CREATE TABLE IF NOT EXISTS users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    password VARCHAR(255) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
