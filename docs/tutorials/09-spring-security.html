Spring Security

SecurityFilterChain, JWT filter, @PreAuthorize, roluri, UserDetailsService

1. Ce este Spring Security?

Spring Security este framework-ul de autentificare și autorizare pentru aplicații Spring. Funcționează printr-un lanț de filtre HTTP care interceptează fiecare request înainte de a ajunge la controller.

  • Autentificare — Cine ești? (verificare username/password, validare JWT)
  • Autorizare — Ce ai voie să faci? (roluri: ADMIN, STUDENT)

3. SecurityConfiguration – configurarea completă

@Configuration
@EnableWebSecurity                   // activează Spring Security
@EnableMethodSecurity                 // activează @PreAuthorize pe metode
public class SecurityConfiguration {

    private final JwtAuthFilter jwtAuthFilter;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            // 1. Dezactivăm CSRF (nu avem sesiuni, folosim JWT)
            .csrf(csrf -> csrf.disable())

            // 2. Fără sesiuni — stateless (JWT are toată informația)
            .sessionManagement(s -> s
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS))

            // 3. Regulile de autorizare
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/**").permitAll()          // login/register: toată lumea
                .requestMatchers("/actuator/**").permitAll()      // health checks: toată lumea
                .requestMatchers("/v3/api-docs/**").permitAll()   // Swagger: toată lumea
                .anyRequest().authenticated()                   // orice altceva: autentificat
            )

            // 4. Adăugăm filtrul JWT ÎNAINTE de filtrul standard de username/password
            .addFilterBefore(jwtAuthFilter,
                UsernamePasswordAuthenticationFilter.class)

            .build();
    }

    @Bean
    public UserDetailsService userDetailsService() {
        return username -> userRepository.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException("User not found"));
    }

    @Bean
    public AuthenticationProvider authenticationProvider() {
        DaoAuthenticationProvider provider = new DaoAuthenticationProvider();
        provider.setUserDetailsService(userDetailsService());
        provider.setPasswordEncoder(passwordEncoder());
        return provider;
    }
}

4. JwtAuthFilter – filtrul personalizat

Extinde OncePerRequestFilter — garantat că rulează o singură dată per request.

@Component
public class JwtAuthFilter extends OncePerRequestFilter {

    private final JwtUtil jwtUtil;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain) throws IOException, ServletException {

        // 1. Citim header-ul Authorization
        String authHeader = request.getHeader("Authorization");

        // 2. Dacă nu există sau nu începe cu "Bearer ", trecem mai departe
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        // 3. Extragem tokenul (fără prefixul "Bearer ")
        String token = authHeader.substring(7);

        // 4. Validăm tokenul
        if (jwtUtil.isTokenValid(token)) {
            String userId = jwtUtil.extractUserId(token);   // ex: "1"
            String role   = jwtUtil.extractRole(token);     // ex: "ADMIN"

            // 5. Creăm obiectul de autentificare
            UsernamePasswordAuthenticationToken auth =
                new UsernamePasswordAuthenticationToken(
                    userId,                  // principal = userId (String)
                    null,                    // credentials = null (nu mai avem nevoie)
                    List.of(new SimpleGrantedAuthority("ROLE_" + role))
                    // authorities: ROLE_ADMIN sau ROLE_STUDENT
                );

            // 6. Stocăm autentificarea în SecurityContext
            // (disponibil în tot request-ul curent)
            SecurityContextHolder.getContext().setAuthentication(auth);
        }

        // 7. Continuăm cu următorul filtru din lanț
        filterChain.doFilter(request, response);
    }
}

De ce ROLE_ prefix? Spring Security adaugă prefixul ROLE_ intern. Dacă în token avem "ADMIN", trebuie să creem autoritatea ca "ROLE_ADMIN". Când folosim hasRole('ADMIN') în @PreAuthorize, Spring compară cu "ROLE_ADMIN" automat. hasAuthority('ROLE_ADMIN') face comparația fără adăugare de prefix.

6. @PreAuthorize – autorizare la nivel de metodă

Necesită @EnableMethodSecurity pe configurare. Evaluează o expresie SpEL înainte de execuția metodei.

// Doar ADMIN poate accesa
@PreAuthorize("hasRole('ADMIN')")
@GetMapping("/all")
public ResponseEntity<Page<UserDto>> getAll() { ... }

// Orice user autentificat
@PreAuthorize("isAuthenticated()")
@GetMapping
public ResponseEntity<UserProfileDto> getOwnProfile() { ... }

// ADMIN sau proprietarul resursei
@PreAuthorize("hasRole('ADMIN') or principal == #userId.toString()")
@GetMapping("/{userId}")
public ResponseEntity<UserProfileDto> getById(@PathVariable Long userId) { ... }
// principal = auth.getName() = userId din token
// #userId = parametrul metodei
Expresie SpELÎnțeles
hasRole('ADMIN')Userul are rolul ADMIN (caută ROLE_ADMIN)
hasAnyRole('ADMIN','MOD')Are rolul ADMIN sau MOD
hasAuthority('ROLE_ADMIN')Are autoritatea exactă ROLE_ADMIN (fără prefix automat)
isAuthenticated()Este autentificat (are token valid)
isAnonymous()Nu este autentificat
principal == #id.toString()Principal-ul (userId din token) egal cu parametrul id
#oauth2.hasScope('read')Are scope-ul OAuth2 (pentru OAuth)

7. UserDetailsService – interfața de autentificare

// Implementată în ApplicationConfiguration
@Bean
public UserDetailsService userDetailsService() {
    return username -> {
        User user = userRepository.findByUsername(username)
            .orElseThrow(() -> new UsernameNotFoundException("User not found: " + username));

        // Returnează un UserDetails — Spring îl folosește la autentificare
        return org.springframework.security.core.userdetails.User
            .withUsername(user.getUsername())
            .password(user.getPassword())        // parola hash BCrypt din DB
            .roles(user.getRole().name())        // adaugă automat ROLE_ prefix
            .disabled(!user.isEnabled())
            .build();
    };
}

Fluxul de login:
1. AuthenticationManager.authenticate(username, password)
2. DaoAuthenticationProvider apelează UserDetailsService.loadUserByUsername(username)
3. Se compară parola din request (text plain) cu hash-ul din DB folosind BCryptPasswordEncoder.matches()
4. Dacă match → autentificare reușită → generăm JWT
5. Dacă nu match → BadCredentialsException

8. PasswordEncoder – BCrypt

// Înregistrare user — parola se hash-uiește
user.setPassword(passwordEncoder.encode(request.getPassword()));
// rezultat: "$2a$10$N6..." (BCrypt cu cost factor 10, salt aleator)

// Login — verificare parolă
// Nu compari direct — encoderele BCrypt includ salt în hash!
passwordEncoder.matches(rawPassword, storedHash); // returnează true/false

// BCrypt adaugă automat salt aleator → același text dă hash-uri diferite
passwordEncoder.encode("pass123"); // "$2a$10$abc..."
passwordEncoder.encode("pass123"); // "$2a$10$xyz..." — DIFERIT!

Nu stoca parole plain text niciodată! BCrypt este algoritm one-way — din hash nu poți recupera parola originală. Cost factor (default 10) determină cât de lent este hash-ul — mai lent = mai greu de forțat brute-force.

9. CSRF – de ce îl dezactivăm?

CSRF (Cross-Site Request Forgery) este un atac unde un site malițios face request-uri în numele tău pe un alt site. Spring Security activează protecția CSRF implicit prin tokens în sesiune.

În aplicații REST cu JWT, dezactivăm CSRF deoarece:

  • Nu avem sesiuni (stateless)
  • Tokenul JWT trebuie trimis explicit în header — atacurile CSRF nu pot accesa header-ul Authorization
  • Browser-ul nu adaugă automat JWT în requests cross-site

12. Întrebări posibile la examen

Q: Ce este Spring Security Filter Chain și cum funcționează?
Spring Security interceptează toate requesturile HTTP printr-un lanț de filtre (Filter Chain). Fiecare filtru face o verificare specifică. Ordinea contează: JwtAuthFilter rulează primul, setează autentificarea în SecurityContext, apoi celelalte filtre verifică regulile de autorizare. Filtrul nostru personalizat este adăugat înainte de UsernamePasswordAuthenticationFilter cu addFilterBefore().
Q: Ce este SecurityContext și de ce e important?
SecurityContext este un holder thread-local care stochează informații despre userul autentificat curent. JwtAuthFilter populează SecurityContext după validarea tokenului. Controller-ul accesează userul prin Authentication auth injectat automat de Spring, sau prin SecurityContextHolder.getContext().getAuthentication(). La finalul fiecărui request, SecurityContext este curățat.
Q: Diferența dintre @PreAuthorize hasRole și hasAuthority?
hasRole('ADMIN') adaugă automat prefixul ROLE_ și caută ROLE_ADMIN în lista de authorities. hasAuthority('ROLE_ADMIN') face comparație exactă fără prefix automat. În proiect, când creăm autoritatea în JwtAuthFilter folosim new SimpleGrantedAuthority("ROLE_" + role), deci ambele forme ar funcționa, dar hasRole este mai concis.
Q: De ce dezactivăm CSRF în REST API-uri cu JWT?
CSRF protejează împotriva atacurilor unde un browser face request-uri automate cu cookie-uri de sesiune. Deoarece folosim JWT în headerul Authorization (nu în cookie), browserul nu trimite automat tokenul în requests cross-site. Atacatorul nu poate accesa headerul Authorization. Deci protecția CSRF este inutilă și ar complica development-ul (ar trebui să trimiți un CSRF token în fiecare request POST/PUT).
Q: Cum funcționează BCrypt și de ce nu poți recupera parola originală?
BCrypt este un algoritm de hashing one-way cu salt aleator. Saltul este inclus în hash-ul final, deci același input dă output-uri diferite. La verificare, BCrypt extrage saltul din hash și re-hash-uiește parola introdusă cu același salt. Dacă match → parola e corectă. Nu poți recupera parola originală din hash — este computațional imposibil (would require brute-forcing billions of combinations). Cost factor 10 înseamnă 2^10 = 1024 iterații de hash.
Q: Ce face @EnableMethodSecurity?
Activează suportul pentru adnotările de securitate la nivel de metodă: @PreAuthorize (evaluat înainte de metodă), @PostAuthorize (evaluat după metodă, poate accesa rezultatul cu #returnObject), @PreFilter și @PostFilter (pentru filtrarea colecțiilor). Fără aceasta, @PreAuthorize este ignorat și oricine poate apela metodele.
Q: Ce diferență este între SessionCreationPolicy.STATELESS și ALWAYS?
STATELESS: Spring Security nu creează niciodată o sesiune HTTP și nu folosește sesiunile existente. Fiecare request trebuie să fie complet independent, cu tot contextul de autentificare (JWT). ALWAYS: Spring creează sesiune la fiecare request autentificat și stochează autentificarea în sesiune. STATELESS este corect pentru REST API-uri cu JWT — nu are sens să stochezi autentificarea în sesiune când o verifici prin JWT la fiecare request.

13. Implementare pas cu pas — Spring Security cu JWT

Prerequisite: spring-boot-starter-security în pom.xml. JwtUtil creat (Tutorial 01).

Pasul 1 — Creează ApplicationConfiguration

Creează config/ApplicationConfiguration.java — definește beans-urile de securitate:

@Configuration
public class ApplicationConfiguration {

    private final UserRepository userRepository;

    // constructor injection...

    @Bean
    public UserDetailsService userDetailsService() {
        return username -> userRepository.findByUsername(username)
                .orElseThrow(() -> new UsernameNotFoundException("User not found: " + username));
        // Spring apelează această metodă la autentificare pentru a încărca userul din DB
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
        // BCrypt: la fiecare apel generează un salt random, deci același plaintext → hash diferit
    }

    @Bean
    public AuthenticationManager authManager(AuthenticationConfiguration config) throws Exception {
        return config.getAuthenticationManager();
    }
}
Pasul 2 — Creează SecurityConfiguration

Creează config/SecurityConfiguration.java:

@Configuration
@EnableWebSecurity
@EnableMethodSecurity   // activează @PreAuthorize pe metode
public class SecurityConfiguration {

    private final JwtAuthFilter jwtAuthFilter;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            // STATELESS: Spring nu creează sesiuni HTTP, fiecare request se autentifică prin JWT
            .csrf(csrf -> csrf.disable())
            // CSRF dezactivat deoarece nu avem sesiuni (JWT este stateless)
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/auth/login", "/auth/register", "/auth/refresh").permitAll()
                .requestMatchers("/api/users/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            // Adaugă filtrul JWT ÎNAINTE de filtrul standard de username/password
            // Astfel JWT e verificat primul la fiecare request
            .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class)
            .build();
    }
}
Pasul 3 — Adaugă @PreAuthorize pe Controller

Cu @EnableMethodSecurity activ, poți proteja metode individuale:

@RestController
@RequestMapping("/api/users")
public class UserController {

    @GetMapping
    @PreAuthorize("hasRole('ADMIN')")          // doar ADMIN
    public ResponseEntity<Page<UserDto>> getAll(Pageable pageable) { ... }

    @DeleteMapping("/{id}")
    @PreAuthorize("hasRole('ADMIN')")
    public ResponseEntity<Void> delete(@PathVariable Long id) { ... }

    @GetMapping("/me")
    @PreAuthorize("isAuthenticated()")         // orice user autentificat
    public ResponseEntity<UserDto> getMe(Authentication auth) { ... }
}

Dacă userul nu are rolul necesar, Spring returnează automat 403 Forbidden.
